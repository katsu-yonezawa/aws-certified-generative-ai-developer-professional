# 運用最適化と監視

## まず結論

- Domain 4 は `安くする`、`速くする`、`見えるようにする`
- 問題文で `トークンコスト`、`レイテンシ`、`スロットル`、`監視` が出たらこのドメイン
- 生成 AI では `品質を落とさずに削る` 発想が重要

## 最適化の順番

1. まず無駄な入力を減らす
2. 次に無駄な出力を減らす
3. その後でキャッシュやモデル切替を考える
4. 最後に容量確保や配備方式を見直す

要点:

- いきなり高価な構成へ進むより、`そもそも送る token を減らせないか` を先に見る
- コストとレイテンシは、かなりの部分が入力長と出力長で決まる

## 1. token とクォータ

### token efficiency

- system prompt を短く保つ
- 不要な履歴を削る
- context を整理する
- 出力長を制限する
- 古い会話は要約してから渡す

### token は費用だけでなくクォータにも効く

- `TPM`
  1 分あたりのトークン使用上限
- `TPD`
  1 日あたりのトークン使用上限
- `RPM`
  1 分あたりのリクエスト上限

### CountTokens

- 入力トークン数を事前に見積もる
- コスト見積もり、長文抑制、上限管理に使う

### max_tokens の見方

- 単なる出力上限ではない
- リクエスト開始時には、入力 token と `max_tokens` を合わせた量がクォータ消費に効く
- 過大な `max_tokens` は、実際の出力が短くても同時実行数を落としやすい

要点:

- `max_tokens を大きめにしておけば安全` は誤り
- コストだけでなくスループット悪化の原因になる

### burndown rate

- モデルによっては、出力 1 token がクォータ上は 1 以上の重みで消費される
- 現行ドキュメントでは、Anthropic Claude 3.7 以降の一部モデルで出力側の burndown rate が `5x`
- 請求は実 token ベースでも、クォータ消費は別計算になることがある

## 2. キャッシュ

### Prompt caching

- 同じ長い先頭部分や共通知識を何度も送るときに有効
- 既定の TTL は `5 分`
- 一部モデルでは `1 時間 TTL` を使える
- 共通部分は先頭へ寄せる

### quota と指標の見方

- `CacheReadInputTokens`
  キャッシュから読んだ入力 token
- `CacheWriteInputTokens`
  キャッシュへ書いた入力 token

要点:

- キャッシュヒットしない設計では効かない
- prefix の一致率と CloudWatch 指標で効き具合を見る

### semantic caching

- 似た質問に対して過去回答を再利用する
- `prompt caching` とは違い、結果再利用の発想

### deterministic hashing

- 完全に同じ入力だけを再利用する
- 誤再利用リスクが低い

### 見分け方

- 長い共通 prompt を何度も送る -> `Prompt caching`
- 似た問い合わせが大量に来る -> `semantic caching`
- 同一入力だけ再利用したい -> `deterministic hashing`

## 3. スループットと容量

### Inference profiles

- アプリケーション単位の利用メトリクスやタグ付けに使う
- 複数リージョンへルーティングする論点ともつながる

### Intelligent prompt routing

- 問い合わせごとに、品質とコストのバランスが良いモデルへ振り分ける発想
- 現行ドキュメントでは prompt router を使って routing 条件や fallback を定義する
- 難しい問いだけ上位モデルへ回したい場面で有効
- 最適化は英語プロンプト中心と整理する

### Cross-Region inference

- 別リージョンの容量も利用して、可用性やスループットを上げる
- どのリージョンへ飛ぶか、データ所在要件をどう扱うかを必ず確認する

### Provisioned Throughput

- 安定した処理枠を確保したいときの選択肢
- 高ボリューム本番や安定遅延が必要な場面で検討する
- 一時的なバースト吸収だけなら、cross-Region inference やアプリ側制御の方が自然なこともある

### Batch inference

- 大量データをまとめて非リアルタイム処理する
- 夜間一括要約、レビュー分類、議事録処理に向く

要点:

- `Batch inference` と `Provisioned Throughput` は別論点
- 一括処理か、常時性能保証かで切る

## 4. レイテンシ最適化

### 有効な打ち手

- ストリーミング
- 並列リクエスト
- 事前計算
- 軽量モデル利用
- キャッシュ
- 出力長の制御

### パラメータ最適化

- `temperature`
  多様性の調整
- `top-k` / `top-p`
  候補集合の絞り方の調整

要点:

- パラメータ調整は効くが、検索設計や入力整理より後
- `速いが雑` と `遅いが高品質` のバランスで見る

## 5. RAG 性能最適化

### 見るべき点

- chunking
- metadata filtering
- hybrid search
- reranking
- index tuning
- query preprocessing

### OpenSearch とフィルタ

- フィルタを後段でかけると、`k` 件より少ない結果しか返らないことがある
- 検索中にフィルタを反映する設計の方が、条件に合う候補を取りやすい

### 正確さを分けて考える

- 検索段階
  recall、precision、filter の影響、topK
- 生成段階
  correctness、completeness、faithfulness、citation precision、citation coverage

要点:

- `検索が悪い` 問題と `生成が逸れる` 問題を混同しない

## 6. 監視

### CloudWatch で見たいもの

- レイテンシ
- エラー率
- スロットル
- `InputTokenCount`
- `OutputTokenCount`
- `CacheReadInputTokens`
- `CacheWriteInputTokens`
- ツール呼び出し成功率
- RAG の取得件数

### model invocation logging

- 推論入力と出力を後から調べる
- 既定では無効
- 有効化したら、保存先の暗号化、保持期間、アクセス制御を設計する

### CloudTrail

- API 操作監査
- 誰が設定を変えたかを追う

### 監視で原因を切り分ける見方

- `レイテンシ悪化`
  モデル、検索、ツール、外部 API のどこが遅いかを見る
- `コスト悪化`
  token 増加、キャッシュ低下、モデル切替漏れを見る
- `品質悪化`
  検索品質、prompt 変更、データ更新、ツール失敗を見る

## 7. 生成 AI 特有の KPI

- hallucination rate
- groundedness
- prompt effectiveness
- answer acceptance rate
- user feedback
- cost per task
- latency per useful answer

要点:

- CPU やメモリだけでは足りない
- `ユーザー価値に結びつく品質指標` を持つ

## 8. ベクトルストア運用

### 見ること

- インデックス状態
- 検索遅延
- 埋め込み品質
- 更新遅延
- 古いデータの残存

### 運用で起きやすいこと

- 文書更新後に古い埋め込みが残る
- チャンク方針変更で検索品質が揺れる
- メタデータ不整合で誤取得が起きる

## 9. アラート設計

- レイテンシ急増
- token 急増
- スロットル増加
- 回答品質低下
- ツール失敗率増加
- 検索ゼロ件率増加

## 10. 引っかけポイント

- コスト問題でまずモデル変更だけを考えない
- `max_tokens` を大きくしすぎる副作用を見落とさない
- レイテンシ問題でまずインフラ増強だけを考えない
- 監視をインフラメトリクスだけで終わらせない
- キャッシュで品質低下リスクを無視しない
- `Prompt caching` と `semantic caching` を混同しない
- `burndown rate` を請求単価と同一視しない
- `CloudTrail` と `model invocation logging` を混同しない

## 11. 判断フロー

1. コストか、速度か、両方か
2. 入力削減か、出力削減か、モデル切替か
3. キャッシュできるか
4. 容量確保が必要か
5. RAG 側の改善余地はあるか
6. 何を KPI として継続監視するか

## Active Recall

- token efficiency で最初に削るものは何か
- `max_tokens` を大きくしすぎると何が起きるか
- Prompt caching の既定 TTL は何分か
- Prompt caching と semantic caching はどう違うか
- `Inference profiles` と `Provisioned Throughput` はどう違うか
- `Intelligent prompt routing` は何のために使うか
- Cross-Region inference は何のために使うか
- RAG の検索品質が低いときに見る要素は何か
- `CloudWatch`、`CloudTrail`、`model invocation logging` は何が違うか
- 生成 AI 特有の KPI には何があるか
- `CacheReadInputTokens` と `CacheWriteInputTokens` は何を見る指標か

### 答えと解説

1. 不要な履歴、冗長な system prompt、長すぎる出力です。まず token の無駄を減らします。
2. 初期クォータ消費が大きくなり、同時実行数を落としやすくなります。コストだけでなくスループットの問題です。
3. `5 分` です。一部モデルでは `1 時間 TTL` も使えます。
4. `Prompt caching` は共通 prefix の再利用、`semantic caching` は似た質問への結果再利用です。前者は入力再利用、後者は結果再利用です。
5. `Inference profiles` はアプリ単位の管理や cross-Region 推論、`Provisioned Throughput` は安定した処理枠の確保です。
6. 難しい問いだけ上位モデルへ回したいときです。平常時の品質とコストの最適化が主目的です。
7. 可用性とスループットを上げるためです。ただし、どのリージョンへ飛ぶかとデータ所在要件は別に確認します。
8. chunking、metadata filtering、hybrid search、reranking、topK、index tuning です。検索段階の改善余地を先に見ます。
9. `CloudWatch` は運用監視、`CloudTrail` は API 操作監査、`model invocation logging` は推論入出力の監査です。
10. hallucination rate、groundedness、acceptance rate、cost per task などです。品質と事業価値を結びます。
11. キャッシュの効き具合です。prefix 設計やキャッシュヒット率の改善に使います。
