# 運用最適化と監視

## まず結論

- Domain 4 は `安くする`、`速くする`、`見えるようにする`
- 問題文で `トークンコスト`、`レイテンシ`、`スロットル`、`監視` が出たらこのドメイン
- 生成 AI では `品質を落とさずに削る` 発想が重要

## 最適化の順番

1. まず無駄な入力を減らす
2. 次に無駄な出力を減らす
3. その後でキャッシュやモデル切替を考える
4. 最後に容量確保や配備方式を見直す

要点:

- いきなり `高性能な構成` を選ぶより、`そもそも送る token を減らせないか` を先に見る
- 生成AIのコストとレイテンシは、かなりの部分が入力長と出力長で決まる
- 試験でも `まず入力整理` を押さえると誤答を外しやすい

## 1. コスト最適化

### token efficiency

- system prompt を短く保つ
- 不要な履歴を削る
- context window を整理する
- 出力長を制限する
- 要約してから渡す

### 覚え方

- `入力を減らす`
- `出力を減らす`
- `大きいモデルを必要なときだけ使う`

### モデル選択の考え方

- 常に最上位モデルを使わない
- 難しい問い合わせだけ高性能モデルへ回す
- 価格対性能で判断する
- 簡単な問い合わせは小型モデルへ寄せる
- `model cascading` や tiered usage を考える

## 2. キャッシュ

### prompt caching

- 同じ長い前置きや共通知識を何度も送るときに有効
- Bedrock の推論コストとレイテンシを下げやすい

### semantic caching

- 似た質問に過去回答を再利用する
- 完全一致でなく意味近似で再利用する

### deterministic hashing

- 完全に同じリクエストだけを再利用する
- 誤再利用リスクが低い

### result fingerprinting と edge caching

- `result fingerprinting`
  応答再利用のキーを安定化させる
- `edge caching`
  ユーザー近くで再利用し、体感レイテンシを下げる

### 見分け方

- 長い共通 prompt を何度も送る -> `prompt caching`
- 似た問い合わせが大量に来る -> `semantic caching`
- 同一入力の再実行を避けたい -> `hashing`
- 再利用判定を安定させたい -> `result fingerprinting`
- グローバル配信で近くから返したい -> `edge caching`

## 3. スループットと容量

### Provisioned Throughput

- 安定した本番性能が必要
- 容量確保が重要
- 予測可能な負荷がある

### batching

- まとめて処理して効率を上げる
- オンラインでなくてもよい処理に向く

### capacity planning

- トークン量
- 同時実行数
- ピーク時間帯
- モデル別負荷

## 4. レイテンシ最適化

### 有効な打ち手

- ストリーミング
- 並列リクエスト
- 事前計算
- 軽量モデル利用
- キャッシュ
- レスポンス長の制御

### パラメータ最適化

- `temperature`
  安定性と多様性の調整
- `top-k` / `top-p`
  候補集合の絞り方の調整

要点:

- パラメータ調整は効くが、検索設計や入力整理より後
- `速いが雑` と `遅いが高品質` のバランスで見る

### 事前計算が効く例

- 定期的に変わる FAQ 要約
- 毎朝更新のダッシュボード説明
- 固定ドキュメントの埋め込み生成

### 速くする施策の効き方

- `ストリーミング`
  体感速度を上げる
- `キャッシュ`
  再計算を減らす
- `軽量モデル`
  推論時間そのものを減らす
- `事前計算`
  実行時の仕事を前倒しする

補足:

- 同じ `レイテンシ改善` でも、何を短くするかが違う
- 体感待ち時間を減らしたいのか、総処理時間を減らしたいのかで打ち手は変わる

## 5. RAG 性能最適化

### 見るべき点

- chunking
- metadata filtering
- hybrid search
- reranking
- index tuning
- query preprocessing

### 典型改善

- チャンクが大きすぎる -> 小さくする
- 断片化しすぎる -> 少し大きくする
- ノイズが多い -> metadata filter
- 取りこぼしが多い -> hybrid search
- 上位順位が悪い -> reranking

## 6. 監視

### CloudWatch で見たいもの

- レイテンシ
- エラー率
- スロットル
- トークン使用量
- キャッシュヒット率
- ツール呼び出し成功率
- RAG の取得件数
- ベクトルストアの遅延
- multi-agent coordination の遅延
- business impact 指標

### Model invocation logging

- リクエストとレスポンスを後から調べる
- 不具合解析、監査、品質比較に使う

### CloudTrail

- API 操作監査
- 誰が設定を変えたかを追う

### 追加で使うもの

- `Cost Explorer`
  コスト傾向を把握する
- `Cost Anomaly Detection`
  コスト急増の検知に使う
- `CloudWatch Synthetics`
  代表ユースケースの継続疎通確認に使う
- `custom dashboards`
  token、品質、業務 KPI をまとめて見る

### 監視で原因を切り分ける見方

- `レイテンシ悪化`
  モデル、検索、ツール、外部 API のどこが遅いかを見る
- `コスト悪化`
  token 増加、キャッシュ低下、モデル切替漏れを見る
- `品質悪化`
  検索品質、prompt 変更、データ更新、ツール失敗を見る

要点:

- 生成AIの障害は、インフラメトリクスだけでは原因が分からない
- `品質`、`コスト`、`処理経路` を同時に観測できる設計が重要

## 7. 生成 AI 特有の KPI

- hallucination rate
- groundedness
- prompt effectiveness
- answer acceptance rate
- user feedback
- cost per task
- latency per useful answer

要点:

- 単なる CPU やメモリだけでは足りない
- `ユーザー価値に結びつく品質指標` を持つ

## 8. ベクトルストア運用

### 見ること

- インデックス状態
- 検索遅延
- 埋め込み品質
- 更新遅延
- 古いデータの残存

### 運用で起きやすいこと

- 文書更新後に古い埋め込みが残る
- チャンク方針変更で検索品質が揺れる
- メタデータ不整合で誤取得が起きる

## 9. アラート設計

- レイテンシ急増
- トークン急増
- スロットル増加
- 回答品質低下
- ツール失敗率増加
- 検索ゼロ件率増加

### anomaly detection

- token burst
- response drift
- cost anomaly

### 追加で見ること

- tool calling の異常増加
- agent coordination の滞留
- synthetic workflow の失敗

## 10. 引っかけポイント

- コスト問題でまずモデル変更だけを考えない
- レイテンシ問題でまずインフラ増強だけを考えない
- 監視をインフラメトリクスだけで終わらせない
- キャッシュで品質低下リスクを無視しない
- RAG の性能問題をモデルのせいにしすぎない
- `result fingerprinting` と `semantic caching` を混同しない
- コスト監視で `Cost Explorer` と anomaly detection を使い分ける

## 11. 判断フロー

1. コストか、速度か、両方か
2. 入力削減か、出力削減か、モデル切替か
3. キャッシュできるか
4. RAG 側の改善余地はあるか
5. 何を KPI として継続監視するか

## Active Recall

- token efficiency で最初に削るものは何か
- prompt caching と semantic caching はどう違うか
- `result fingerprinting` と edge caching は何のために使うか
- Provisioned Throughput はどんな要件で使うか
- レイテンシ改善で有効な打ち手は何か
- `temperature`、`top-k`、`top-p` はどこで効くか
- RAG の検索品質が低いときに見る要素は何か
- CloudWatch、CloudTrail、invocation logging は何が違うか
- `Cost Explorer`、`Cost Anomaly Detection`、`CloudWatch Synthetics` は何を見るか
- 生成 AI 特有の KPI には何があるか

### 答えと解説

1. 不要な履歴、冗長な system prompt、長すぎる出力です。まず token の無駄を減らします。
2. `prompt caching` は共通 prompt の再利用、`semantic caching` は似た質問への回答再利用です。前者は入力再利用、後者は結果再利用です。
3. `result fingerprinting` は再利用判定を安定させ、`edge caching` はユーザー近くで返して体感速度を上げます。キー設計の論点か、配置の論点かで見分けます。
4. 安定した性能と容量確保が必要な本番です。負荷が読めるときに強いです。
5. ストリーミング、並列化、事前計算、軽量モデル、キャッシュです。単純なスケールアップより先に検討します。
6. `temperature`、`top-k`、`top-p` は応答のばらつきや探索範囲に効きます。品質問題の第一候補ではなく、最後の微調整として見ると安定します。
7. chunking、metadata filtering、hybrid search、reranking、index tuning です。検索品質は前段で大きく変わります。
8. `CloudWatch` は運用監視、`CloudTrail` は API 操作監査、`invocation logging` は推論入出力の分析です。
9. `Cost Explorer` は傾向把握、`Cost Anomaly Detection` は急増検知、`CloudWatch Synthetics` は代表フローの疎通確認です。コスト把握、異常検知、体験監視で役割が違います。
10. hallucination rate、groundedness、acceptance rate、cost per task などです。品質と事業価値を結びます。
