# 評価とトラブルシューティング

## まず結論

- Domain 5 は `よい回答かをどう測るか` と `壊れたときにどこから切るか`
- 生成 AI では、従来 ML の精度だけでは足りず、`関連性`、`忠実性`、`一貫性`、`安全性` まで見る
- 問題文で `A/B テスト`、`評価基盤`、`hallucination`、`トラブルシュート` が出たらこのドメイン

## 評価を改善につなげる型

1. 良い回答の定義を決める
2. 代表入力を集める
3. 指標としきい値を決める
4. 変更前後を同条件で比べる
5. 不合格なら原因を層ごとに切る

要点:

- 評価は `点数を出すこと` が目的ではなく、`次に何を直すか決めること` が目的
- そのため `悪い` だけでなく、`検索が悪い`、`形式が壊れる`、`安全性が足りない` のように原因へ戻せる設計が必要
- 試験でも、評価基盤は改善ループの一部として理解すると整理しやすい

## 1. 評価の基本

### 見る軸

- relevance
- factual accuracy
- consistency
- fluency
- safety
- business usefulness

### オフライン評価

- golden dataset を作る
- 同じ入力でモデルや prompt を比較する
- 回帰テストに向く

### オンライン評価

- A/B テスト
- canary
- ユーザーフィードバック
- 業務 KPI 監視

### ユーザー中心の評価

- feedback interface を用意する
- rating system で良し悪しを集める
- annotation workflow で原因付きラベルを残す

要点:

- `悪い` だけでは改善できない
- `どこが悪いか` を残せる入力欄や分類が重要

## 2. Bedrock Evaluations

- モデル比較
- prompt 比較
- LLM-as-a-judge
- human-based evaluation

### 使い分け

- 大量比較を速く回す -> `LLM-as-a-judge`
- 主観やブランド文体も見たい -> `human-based`
- 本番前の定量比較 -> `Bedrock Evaluations`

## 3. 評価設計の考え方

### まず固定するもの

- 何を良い回答とみなすか
- どの入力群で試すか
- どの指標で比べるか
- どのしきい値で採用するか

### よくある評価セット

- 正常系
- 境界値
- 悪意ある入力
- 長文入力
- 検索が必要な質問
- ツール失敗時の質問

### quality gates

- 新モデルや新 prompt の採用条件を先に決める
- 回帰テストを通らない更新は通さない
- 品質ゲートは `主観` ではなく `しきい値` で置く

## 4. RAG 評価

### 分けて見る

- 取得できているか
- 取得結果が関連しているか
- 回答が根拠に忠実か

### 失敗パターン

- retrieval failure
- ranking failure
- chunking failure
- grounding failure
- answer synthesis failure

### 覚え方

- `拾えていない`
- `拾っているが順位が悪い`
- `拾っているのに答えが逸れる`

### RAG を分解して見る理由

- `検索できていない` 問題と `生成が逸れている` 問題では打ち手が違う
- 前者は chunking、metadata、reranking を見直し、後者は prompt や grounding を見直す
- ここを混同すると、誤ってモデル変更や fine-tuning に走りやすい

## 4.1 Agent 評価

- task completion rate
- tool usage effectiveness
- reasoning quality in multi-step workflows
- `AgentCore Evaluations`

要点:

- Agent は `答えがよいか` だけでなく `完了できたか` を測る
- ツール選択ミスと推論品質を分けて見る

## 5. A/B テストとカナリア

### A/B テスト

- モデル、prompt、RAG 設計の比較に使う
- 十分なサンプルと評価指標が必要

### カナリア

- 新モデルや新 prompt を小さく流す
- 品質悪化やコスト急増を早期検知する

### ロールバック

- 品質低下が見えたらすぐ戻せるようにする
- prompt もモデルもバージョン化しておく

### デプロイ検証

- synthetic user workflows を流す
- hallucination rate と semantic drift を見る
- 自動 quality check を通してから広げる

## 6. トラブルシュートの切り分け順

1. 入力
2. prompt
3. モデル
4. 検索
5. ツール
6. 出力フォーマット
7. 監視設定

### 理由

- 生成 AI の失敗は 1 箇所ではなく複数原因が多い
- まず再現条件を固定し、上流から切る

### 失敗を1つの原因に決めつけない

- prompt を変えた直後に品質が落ちても、同時に検索データ更新が入っていることがある
- モデル変更の影響に見えて、実際は tool timeout が原因のこともある
- そのため、`入力集合`、`モデル版`、`prompt 版`、`データ版` をそろえて比較することが重要

## 7. よくある不具合と打ち手

### 回答が根拠にない

- RAG を導入または強化する
- grounding を強くする
- source attribution を確認する

### 回答が不安定

- prompt を固定する
- 変数の渡し方を見直す
- temperature などのパラメータを見直す

### JSON が壊れる

- structured outputs
- JSON Schema
- 後段バリデーション

### 文脈が切れる

- 履歴を整理する
- 要約メモリにする
- 長文を分割する
- context window overflow を疑う
- dynamic chunking を試す

### 検索精度が悪い

- chunking
- embeddings
- metadata filtering
- reranking

### ツール呼び出しが失敗する

- 引数検証
- タイムアウト
- 再試行
- 権限確認

## 8. ログと可観測性の使い分け

### CloudWatch Logs

- 実行ログ
- エラー詳細
- prompt 比較の材料

### X-Ray

- 分散トレーシング
- 多段処理の遅延把握

### invocation logging

- 推論入出力の詳細確認

### golden dataset

- 継続品質確認
- 更新時の回帰検知

### stakeholder reporting

- 可視化で比較結果を伝える
- モデル差、prompt 差、RAG 差を同じ指標で並べる
- 品質、速度、コストをまとめて見せる

## 9. デプロイ前後で見ること

### デプロイ前

- 評価セットで比較
- 安全性テスト
- コスト見積もり
- latency benchmark

### デプロイ後

- feedback 収集
- drift 監視
- token 監視
- answer quality 監視
- rating と annotation の継続収集
- synthetic workflow の定期実行

## 10. 引っかけポイント

- 生成 AI を単一精度指標だけで評価しない
- RAG の失敗をモデル性能だけの問題にしない
- prompt 問題を検索問題と混同しない
- JSON 破損を後段アプリのせいにしすぎない
- 本番更新でカナリアや回帰テストを省かない

## 11. 判断フロー

1. 何を評価したいか
2. オフラインか、オンラインか
3. どの層が壊れているか
4. 再現条件を固定できるか
5. ロールバック可能か

## Active Recall

- 生成 AI で見るべき評価軸は何か
- LLM-as-a-judge と human evaluation はどう使い分けるか
- feedback interface、rating system、annotation workflow は何のために入れるか
- RAG 評価は何段階に分けて考えるか
- Agent 評価では何を見るか
- A/B テストとカナリアの違いは何か
- quality gate と synthetic user workflow は何を守るか
- トラブルシュートで最初に固定すべきものは何か
- JSON 破損、検索不良、回答不安定の第一候補の打ち手は何か

### 答えと解説

1. relevance、factual accuracy、consistency、fluency、安全性、業務有効性です。従来 ML より多面的です。
2. 大量の比較を速く回すなら `LLM-as-a-judge`、主観品質やブランド適合を見るなら `human evaluation` です。
3. feedback interface は回収口、rating system は定量化、annotation workflow は原因付きで蓄積する仕組みです。改善の解像度を上げるために使います。
4. `取得`、`順位`、`回答忠実性` に分けます。検索と生成を分けて考えます。
5. Agent では task completion rate、tool usage effectiveness、reasoning quality を見ます。回答文だけ見ても不十分です。
6. A/B は比較実験、カナリアは小規模導入です。前者は優劣確認、後者は安全な展開です。
7. quality gate は更新前の足切り、synthetic user workflow は代表操作の自動検証です。品質悪化を本番へ出さないために使います。
8. 入力、prompt、モデル条件、データセットです。再現できないと改善も検証もできません。
9. JSON 破損は structured outputs、検索不良は chunking と reranking、回答不安定は prompt とパラメータの固定が第一候補です。
