# 評価とトラブルシューティング

## まず結論

- Domain 5 は `よい回答かをどう測るか` と `壊れたときにどこから切るか`
- 生成 AI では、従来 ML の精度だけでなく `関連性`、`忠実性`、`完全性`、`安全性` まで見る
- 問題文で `A/B テスト`、`ground truth`、`hallucination`、`RAG 評価`、`トラブルシュート` が出たらこのドメイン

## 評価を改善につなげる型

1. 良い回答の定義を決める
2. 代表入力と ground truth を集める
3. 指標としきい値を決める
4. 変更前後を同条件で比べる
5. 不合格なら原因を層ごとに切る

要点:

- 評価は `点数を出すこと` ではなく `次に何を直すか決めること` が目的
- `悪い` だけでなく、`検索が悪い`、`生成が逸れる`、`形式が壊れる`、`安全性が足りない` のように原因へ戻せる形で見る

## 1. 評価の基本

### 見る軸

- relevance
- correctness
- completeness
- faithfulness
- consistency
- safety
- business usefulness

### オフライン評価

- golden dataset や ground truth dataset を作る
- 同じ入力でモデル、prompt、RAG 設計を比較する
- 回帰テストに向く

### オンライン評価

- A/B テスト
- canary
- ユーザーフィードバック
- 業務 KPI 監視

### ユーザー中心の評価

- feedback interface を用意する
- rating だけでなく、`何が悪かったか` を残せるようにする
- annotation workflow で原因付きラベルを集める

## 2. Bedrock Evaluations

- モデル比較
- prompt 比較
- RAG 比較
- `LLM-as-a-judge`
- `human-based evaluation`

### 使い分け

- `LLM-as-a-judge`
  大量比較を速く回したい
- `human-based evaluation`
  ブランド文体や微妙な UX 差も見たい

### quality gate の置き方

- 採用条件を先に決める
- 変更前後で同じデータセットを流す
- 回帰したら採用しない

## 3. 評価設計の考え方

### まず固定するもの

- 何を良い回答とみなすか
- どの入力群で試すか
- どの指標で比べるか
- どのしきい値で採用するか

### よくある評価セット

- 正常系
- 境界値
- 悪意ある入力
- 長文入力
- 検索が必要な質問
- ツール失敗時の質問

要点:

- `良い` の定義が曖昧だと、評価結果を次の実装に戻せない

## 4. RAG 評価

### retrieve only と retrieve and generate を分ける

- `retrieve only`
  検索そのものの品質を見る
- `retrieve and generate`
  最終回答の品質を見る

### retrieve only で見るもの

- context relevance
- context coverage
- recall
- precision
- ranking quality

### retrieve and generate で見るもの

- correctness
- completeness
- helpfulness
- faithfulness
- citation precision
- citation coverage

### 失敗パターン

- retrieval failure
- ranking failure
- chunking failure
- grounding failure
- answer synthesis failure

### RAG を分解して見る理由

- `検索できていない` 問題と `生成が逸れている` 問題では打ち手が違う
- 前者は chunking、metadata、filter、reranking、topK を見直す
- 後者は prompt、grounding、出力制約、Guardrails を見直す

## 5. Agent 評価

- task completion rate
- tool usage effectiveness
- reasoning quality in multi-step workflows
- failure recovery

要点:

- Agent は `答えがよいか` だけでなく `完了できたか` を測る
- ツール選択ミスと推論品質を分けて見る

## 6. A/B テストとカナリア

### A/B テスト

- モデル、prompt、RAG 設計の比較に使う
- 十分なサンプルと同じ評価指標が必要

### カナリア

- 新モデルや新 prompt を小さく流す
- 品質悪化やコスト急増を早期検知する

### ロールバック

- 品質低下が見えたらすぐ戻せるようにする
- prompt、モデル、Flow、Agent は版管理しておく

## 7. トラブルシュートの切り分け順

1. 入力
2. prompt
3. モデル
4. 検索
5. ツール
6. 出力フォーマット
7. 監視設定

### 理由

- 生成 AI の失敗は 1 箇所ではなく複数原因が多い
- まず再現条件を固定し、上流から切る

### 失敗を 1 つの原因に決めつけない

- prompt を変えた直後の品質低下でも、同時に検索データ更新が入っていることがある
- モデル変更の影響に見えて、実際は tool timeout が原因のこともある
- そのため、`入力集合`、`モデル版`、`prompt 版`、`データ版` をそろえて比較する

## 8. よくある不具合と打ち手

### 回答が根拠にない

- RAG を導入または強化する
- grounding を強くする
- source attribution を確認する

### 回答が不安定

- prompt を固定する
- 変数の渡し方を見直す
- `temperature` などのパラメータを見直す

### JSON が壊れる

- structured outputs
- JSON Schema
- 後段バリデーション

### 文脈が切れる

- 履歴を整理する
- 要約メモリにする
- 長文を分割する
- context overflow を疑う

### 検索精度が悪い

- chunking
- embeddings
- metadata filtering
- topK
- reranking

### ツール呼び出しが失敗する

- 引数検証
- タイムアウト
- 再試行
- 権限確認

## 9. ログと可観測性の使い分け

### CloudWatch Logs

- 実行ログ
- エラー詳細
- 比較評価の材料

### Trace

- Agent や Flow の内部手順を見る
- 多段処理のどこで詰まったかを追う

### model invocation logging

- 推論入出力の確認
- 既定では無効

### golden dataset

- 継続品質確認
- 更新時の回帰検知

## 10. デプロイ前後で見ること

### デプロイ前

- 評価セットで比較
- 安全性テスト
- コスト見積もり
- latency benchmark

### デプロイ後

- feedback 収集
- KPI 監視
- 回帰検知
- 失敗例の再ラベル

## 11. 引っかけポイント

- `評価` と `Guardrails` を混同しない
- RAG の失敗を全部モデルのせいにしない
- `retrieve only` と `retrieve and generate` を混ぜて評価しない
- 変更前後でデータセットを変えない
- `Trace` と `CloudTrail` を混同しない

## 12. 判断フロー

1. 何を良い回答と定義するか
2. ground truth はあるか
3. 検索段階か生成段階か
4. オフライン評価かオンライン評価か
5. 回帰検知とロールバック手段はあるか

## Active Recall

- 良い評価セットに必要なものは何か
- `LLM-as-a-judge` と human-based evaluation はどう使い分けるか
- `retrieve only` と `retrieve and generate` はどう違うか
- RAG の検索段階で見る指標は何か
- 最終回答段階で見る指標は何か
- Agent 評価で追加で見るべきものは何か
- トラブルシュートで最初に固定すべきものは何か
- `Trace` と `model invocation logging` は何が違うか
- 回帰テストで同じデータセットを使う理由は何か
- 品質低下が見えたら何をすぐ戻せるようにしておくべきか

### 答えと解説

1. 代表入力、ground truth、評価指標、しきい値です。良し悪しの定義がないと改善につながりません。
2. `LLM-as-a-judge` は大量比較を速く回したいとき、human-based evaluation は微妙な品質差や UX 差も見たいときです。
3. `retrieve only` は検索品質、`retrieve and generate` は最終回答品質です。段階を分けないと打ち手がずれます。
4. context relevance、context coverage、recall、precision、ranking quality です。根拠を拾えているかを見ます。
5. correctness、completeness、helpfulness、faithfulness、citation precision、citation coverage です。回答の中身を見る段階です。
6. task completion rate、tool usage effectiveness、failure recovery です。`答えが自然か` だけでは足りません。
7. 入力集合、モデル版、prompt 版、データ版です。再現条件をそろえないと原因切り分けが崩れます。
8. `Trace` は内部手順、`model invocation logging` は推論入出力です。見る対象が違います。
9. 変更前後の純粋比較をするためです。入力条件まで変わると回帰かどうか判断できません。
10. prompt、モデル、Flow、Agent などの版です。悪化時にすぐ戻せるようにします。
