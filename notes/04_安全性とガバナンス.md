# 安全性とガバナンス

## まず結論

- Domain 3 は `危ない入力を止める`、`危ない出力を止める`、`データを守る`、`後から説明できるようにする`
- `Guardrails` だけで終わらず、`IAM`、`PrivateLink`、`監査`、`Organizations`、`Responsible AI` まで問われる
- 問題文に `PII`、`規制`、`監査`、`prompt injection`、`説明責任` が出たらこのドメイン

## 安全対策を層で分ける

1. 入力で止める
2. モデル実行時に止める
3. 出力で止める
4. 権限と経路を絞る
5. ログを残して後から説明できるようにする

要点:

- 安全性は `どの層で守るか` で整理すると強い
- 正解になりやすいのは `実行時制御 + 権限 + 監査` の組み合わせ

## 1. 入出力の安全制御

### まず入れるもの

- `Guardrails`
- 入力検証
- 出力検証
- 構造化出力
- ログとアラート

### Guardrails で見ること

- `content filters`
- `denied topics`
- `word filters`
- `sensitive information filters`
- `contextual grounding checks`
- `automated reasoning checks`
- `prompt attack detection`

### 構造化出力

- JSON Schema や structured outputs を使う
- free-form 文字列より誤解釈が減る
- text-to-SQL や更新系処理では特に有効

### ハルシネーション低減

- 第一候補は `Knowledge Bases` などの grounding
- 必要に応じて再検索、照合、出典提示を足す
- `モデルを大きくする` より `根拠を与える` 方が自然なことが多い

## 2. Guardrails の注意点

### contextual grounding checks

- 参照ソースとユーザー質問を前提に、回答が根拠に基づくかを見る
- 要約、言い換え、質問応答に向く
- Conversational QA や一般チャットをそのまま対象にする機能ではない

### automated reasoning checks

- 定義したルールに対する論理検証に向く
- 強力だが、これだけで prompt injection を防ぐ前提にはしない
- 英語中心、ストリーミング非対応などの制約を確認し、他の制御と合わせて使う

### Guardrails は万能ではない

- `Guardrails` は実行時制御の中心だが、権限制御、閉域、監査証跡までは単独では担わない
- したがって、`IAM`、`PrivateLink`、`CloudTrail`、human review と組み合わせる

## 3. prompt injection / jailbreak 対策

### 典型対策

- システム命令とユーザー入力を分離
- 外部文書をそのまま命令として扱わない
- ツール呼び出し前に許可判定を入れる
- 危険入力の検知とブロックを入れる
- 高リスク処理は human review を入れる

### defense in depth

- 前段: 入力サニタイズ、分類、ルール判定
- 中段: Guardrails、モデル側制御
- 後段: 出力検証、ポリシーフィルタ、監査

要点:

- 1 個の機能で全部守ろうとしない
- 多層防御で `防止`、`縮小`、`追跡` を分けて考える

## 4. データセキュリティとプライバシー

### 最小セット

- IAM 最小権限
- KMS 暗号化
- VPC endpoints / PrivateLink
- S3 保持ポリシー
- 保持期間管理
- CloudTrail 監査

### Bedrock のデータ保護

- Bedrock は既定でプロンプトや補完結果を保存しない
- ただし `model invocation logging` を有効化した場合は、保存先、暗号化、保持期間、アクセス制御を利用者側で設計する

### PII 対応

- `Amazon Comprehend`
  入力テキストの PII 検出に使う
- `Amazon Macie`
  S3 内の機微データ検出に使う
- `Guardrails`
  LLM の入出力側での PII 抑止に使う

## 5. ネットワーク分離

### 覚えるもの

- `VPC endpoints`
- `PrivateLink`
- `network isolation`

### 試験での見分け方

- インターネットを通したくない
- 閉域で Bedrock や他サービスへ接続したい
- 機微データの送信経路を限定したい

これらが見えたら `VPC endpoint / PrivateLink`

## 6. ガバナンスとコンプライアンス

### 追跡すべきもの

- どのデータを使ったか
- どのモデルを使ったか
- どのプロンプトで答えたか
- どの判断がなされたか
- 誰が変更したか

### 何を見る仕組みかを分ける

- `Trace`
  エージェントや Flow の内部手順
- `CloudTrail`
  AWS API 操作の監査証跡
- `model invocation logging`
  推論入出力の記録

### 代表機能

- `Model Cards`
  制約、用途、評価、リスクの文書化
- `Lineage`
  データ、ジョブ、モデルの由来追跡
- `Glue Data Catalog`
  データソース登録とメタデータ管理
- `CloudTrail`
  API 操作監査
- `CloudWatch Logs`
  実行ログや判断ログ

### データ由来

- RAG では `どの文書を根拠にしたか` を残す
- 出力に source attribution が必要なケースがある
- 監査要件があるなら `答え` だけでなく `根拠` と `判断経路` を追えるようにする

## 7. 組織単位の制御

### Organizations と Bedrock policies

- `SCP` は最大権限制御
- Bedrock policies で、組織単位の統制をかける論点がある

### Guardrails enforcement

- 組織全体で Guardrails 適用を強制したいときの論点
- 現時点では `preview` と整理する
- 強制する guardrail は `version` を使い、`DRAFT` を指定しない
- 自動推論ポリシーのように、そのまま強制対象へ置けないものがある点にも注意する

要点:

- `SCP で API を絞る話` と `Guardrails を強制する話` は別
- 組織統制の設問では、この 2 つを分けて考える

## 8. Responsible AI

### 重要観点

- 透明性
- 公平性
- 安全性
- 不確実性の扱い
- 制約の明示

### 試験で問われる形

- ユーザーに根拠や制約を示したい
- バイアスや偏りを継続評価したい
- 説明責任のために判断の痕跡を残したい

### 実装寄りの答え

- source attribution
- confidence や不確実性の可視化
- fairness evaluation
- model card での制約明示
- 継続改善の見直し軸として `Generative AI Lens`

### 継続改善の見方

- モデル差し替え、prompt 変更、Guardrails 更新、chunking 変更のたびに同じデータセットで回帰確認する
- `Well-Architected Framework` の `Generative AI Lens` は、運用、セキュリティ、信頼性、性能、コストの見直しに使いやすい
- 資料を見るときは `initial publication` と `publication date` を混同しない

## 9. SageMaker 側の論点

### Clarify

- バイアス、公平性、説明可能性
- Responsible AI の文脈で出やすい

### Model Cards

- モデルの説明書
- 規制対応や監査説明で効く

### Model Monitor

- 本番の品質やドリフト監視
- safety drift や bias drift にもつながる

## 10. 引っかけポイント

- `Guardrails` だけでセキュリティ全体を解いた気にならない
- PII 検出と PII 抑止を混同しない
- `Trace` と `CloudTrail` を混同しない
- ハルシネーション対策で fine-tuning を第一候補にしない
- `Guardrails enforcement` を一般提供済み前提で読まない
- 監査要件なのに根拠文書や判断ログを残さない

## 11. 判断フロー

1. 何を防ぎたいか
2. 入力側か、出力側か、両方か
3. データ経路をどう閉じるか
4. 何を監査証跡として残すか
5. 組織統制が必要か
6. 説明責任と公平性をどう担保するか

## Active Recall

- Guardrails で何を制御できるか
- `contextual grounding` と `automated reasoning` は何が違うか
- ハルシネーションを減らす第一候補は何か
- Comprehend、Macie、Guardrails はどう使い分けるか
- prompt injection 対策で重要な考え方は何か
- `PrivateLink` が必要な場面は何か
- `Trace`、`CloudTrail`、`model invocation logging` は何を残すか
- `SCP` と Guardrails enforcement はどう違うか
- Guardrails enforcement で `version` を使う理由は何か
- Responsible AI を実装で表すと何になるか

### 答えと解説

1. 禁止トピック、単語フィルタ、PII、grounding、論理検証、prompt attack detection などです。入力と出力の両方を制御できます。
2. `contextual grounding` は根拠との整合性、`automated reasoning` は定義したルールとの論理整合性を見る仕組みです。
3. `Knowledge Bases` などで根拠に基づかせることです。必要に応じて再検索や照合を足します。
4. `Comprehend` は入力の PII 検出、`Macie` は S3 の機微データ検出、`Guardrails` は LLM 入出力の抑止です。
5. 多層防御です。入力検証、モデル制御、出力検証、権限制御、監査を重ねます。
6. インターネットを通したくない、送信経路を閉じたい、閉域接続が必要な場面です。
7. `Trace` は内部手順、`CloudTrail` は API 操作、`model invocation logging` は推論入出力です。残す対象が違います。
8. `SCP` は API や操作の上限制御、Guardrails enforcement は Guardrails 適用の統制です。権限制御か、安全ポリシー適用かで違います。
9. 作業中の `DRAFT` ではなく、固定された `version` を強制対象にするためです。
10. source attribution、公平性評価、model card、不確実性の可視化などです。理念ではなく実装に落とします。
