# 安全性とガバナンス

## まず結論

- Domain 3 は `危ない入力を止める`、`危ない出力を止める`、`データを守る`、`説明責任を持つ`
- Bedrock の `Guardrails` だけで終わらず、`IAM`、`VPC`、`監査`、`Responsible AI` まで問われる
- 問題文に `PII`、`規制`、`監査`、`prompt injection` が出たらこのドメイン

## 安全対策を層で分ける

1. 入力で止める
2. モデル実行時に止める
3. 出力で止める
4. 権限と経路を絞る
5. ログを残して後から説明できるようにする

要点:

- 安全性の論点は、1つのサービス名で解くより `どの層で守るか` で整理すると強い
- `Guardrails` は強いが、権限制御、閉域、監査証跡までは単独では担わない
- 試験で正解になりやすいのは `実行時制御 + 権限 + 監査` の組み合わせ

## 1. 入出力の安全制御

### まず入れるもの

- `Guardrails`
- 入力検証
- 出力検証
- 構造化出力の強制
- ログとアラート

### Guardrails で見ること

- 禁止トピック
- 単語フィルタ
- PII フィルタ
- contextual grounding
- 入力と出力の両方の制御

### 構造化出力

- JSON Schema や structured outputs を使う
- free-form 文字列より誤解釈が減る
- text-to-SQL など高リスク処理では特に有効

### ハルシネーション低減

- 第一候補は `Knowledge Bases` などの grounding
- 必要に応じて信頼度スコア、再検索、照合を足す
- `モデルを大きくする` より `根拠を与える` の方が自然なことが多い

## 2. prompt injection / jailbreak 対策

### 典型対策

- システム命令とユーザー入力を分離
- 外部文書をそのまま命令として扱わない
- ツール呼び出し前に許可判定を入れる
- 危険入力の検知とブロックを入れる
- 高リスク処理は human review を入れる

### defense in depth

- 前段: 入力サニタイズ、分類、ルール判定
- 中段: Guardrails、モデル側制御
- 後段: 出力検証、ポリシーフィルタ、監査

要点:

- 1 個の機能で全部守ろうとしない
- 多層防御で設計する

### なぜ多層防御が必要か

- 入力で防げないものが出力で表面化することがある
- Guardrails で止まらないリスクも、IAM や human review で被害を小さくできる
- 監査ログがないと、事故後に `何が起きたか` を説明できない

補足:

- つまり安全性は `防止`、`縮小`、`追跡` の3つで考えると漏れにくい

## 3. データセキュリティとプライバシー

### 最小セット

- IAM 最小権限
- KMS 暗号化
- VPC endpoints / PrivateLink
- `Lake Formation` による粒度の細かいデータアクセス制御
- S3 保持ポリシー
- S3 Lifecycle による保持期間管理
- CloudTrail 監査

### PII 対応

- `Amazon Comprehend`
  PII 検出や匿名化の前段処理に使う
- `Amazon Macie`
  S3 内の機微データ検出に使う
- `Guardrails`
  出力側の PII 抑制に使う

### どう切り分けるか

- 入力文の PII 検出 -> `Comprehend`
- S3 データ湖の機微データ発見 -> `Macie`
- LLM 出力の PII 抑止 -> `Guardrails`

## 4. ネットワーク分離

### 覚えるもの

- `VPC endpoints`
- `PrivateLink`
- `network isolation`

### 試験での見分け方

- インターネットを通したくない
- 閉域で Bedrock や他サービスへ接続したい
- 機微データの送信経路を限定したい

これらが見えたら `VPC endpoint / PrivateLink`

## 5. ガバナンスとコンプライアンス

### 追跡すべきもの

- どのデータを使ったか
- どのモデルを使ったか
- どのプロンプトで答えたか
- どの判断がなされたか
- 誰が変更したか

### 代表機能

- `Model Cards`
  制約、用途、評価、リスクの文書化
- `Lineage`
  データ、ジョブ、モデルの由来追跡
- `Glue Data Catalog`
  データソース登録とメタデータ管理
- `metadata tagging`
  出力に根拠属性を持たせる
- `CloudTrail`
  API 操作監査
- `CloudWatch Logs`
  判断ログ、業務ログ

### データ由来

- RAG では `どの文書を根拠にしたか` を残す
- 出力に source attribution が必要なケースがある
- 監査要件があるなら `答え` だけでなく `根拠` を追えるようにする

### 継続ガバナンス

- misuse、drift、policy violations を継続検知する
- `token-level redaction` や出力ポリシーフィルタで漏えいを減らす
- response logging と automated alerting を組み合わせる
- 監査では `その場で防いだか` と `後から説明できるか` の両方が必要

### 監査要件で見落としやすい点

- `答え` だけでなく `根拠` と `判断経路` を残す
- 誰が設定を変えたかを `CloudTrail` で追えるようにする
- モデルや prompt を更新した履歴を版として残す

要点:

- 規制対応の設問では、単なる暗号化やフィルタだけでは足りない
- `再現できること` と `説明できること` が求められる

## 6. Responsible AI

### 重要観点

- 透明性
- 公平性
- 安全性
- 不確実性の扱い
- 制約の明示

### 試験で問われる形

- ユーザーに根拠や制約を示したい
- バイアスや偏りを継続評価したい
- 説明責任のために意思決定の痕跡を残したい

### 実装寄りの答え

- source attribution
- confidence / uncertainty の可視化
- reasoning trace や agent tracing
- reasoning displays
- fairness evaluation
- model card で制約明示

### 試験での補足

- `confidence metrics` は不確実性の可視化に使う
- fairness は理念ではなく `継続評価` と `比較手順` に落とす
- `Prompt Management` や `Prompt Flows` を使った systematic A/B が責任ある改善につながる

## 7. SageMaker 側の論点

### Clarify

- バイアス、公平性、説明可能性
- Responsible AI の文脈で出やすい

### Model Cards

- モデルの説明書
- 規制対応や監査説明で効く

### Model Monitor

- 本番の品質やドリフト監視
- safety drift や bias drift にもつながる

## 8. 引っかけポイント

- `Guardrails` だけでセキュリティ全体を解いた気にならない
- PII 検出と PII 抑止を混同しない
- ハルシネーション対策で fine-tuning を第一候補にしない
- 監査要件なのに根拠文書や判断ログを残さない
- Responsible AI を理念だけで覚えず、実装機能に結びつける
- `Lake Formation` や metadata tagging を見落としてデータ統制を弱くしない
- token-level redaction と PII 検出を同じものと考えない

## 9. 判断フロー

1. 何を防ぎたいか
2. 入力側か、出力側か、両方か
3. データ経路をどう閉じるか
4. 何を監査証跡として残すか
5. 説明責任と公平性をどう担保するか

## Active Recall

- Guardrails で何を制御できるか
- ハルシネーションを減らす第一候補は何か
- Comprehend、Macie、Guardrails はどう使い分けるか
- prompt injection 対策で重要な考え方は何か
- VPC endpoint / PrivateLink が必要な場面は何か
- `Lake Formation` はどの論点で効くか
- Model Cards、Lineage、CloudTrail は何を残すか
- metadata tagging は何のために使うか
- policy violations、token-level redaction、response logging は何を守るか
- Responsible AI を実装で表すと何になるか

### 答えと解説

1. 禁止トピック、単語フィルタ、PII、grounding などです。入力と出力の両方を制御できます。
2. `Knowledge Bases` などで根拠に基づかせることです。必要に応じて再検索や検証を足します。
3. `Comprehend` は入力の PII 検出、`Macie` は S3 の機微データ検出、`Guardrails` は出力側の抑止です。
4. 多層防御です。入力検証、モデル制御、出力検証、権限制御を重ねます。
5. 閉域接続や送信経路制御が必要なときです。機微データや規制要件で出やすいです。
6. `Lake Formation` はデータアクセスを粒度細かく制御したいときに効きます。閉域だけでなく、誰がどの列やテーブルへ触れるかを絞る論点です。
7. `Model Cards` は制約や用途、`Lineage` は由来、`CloudTrail` は API 操作です。役割が違います。
8. metadata tagging は、どのデータ源から来た答えかを系統的に残すために使います。source attribution や監査説明に効きます。
9. policy violations の検知、token-level redaction、response logging は、規制違反や漏えいを継続的に抑えるための仕組みです。防止、記録、是正を分けて考えます。
10. source attribution、不確実性の可視化、公平性評価、制約文書化です。理念をログと UI に落とすことが重要です。
