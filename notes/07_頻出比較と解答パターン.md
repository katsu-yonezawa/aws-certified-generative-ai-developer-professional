# 頻出比較と解答パターン

## まず結論

- AIP-C01 は `比較問題` が多い
- サービス名を丸暗記するより、`何を変える機能か` を見分ける
- 最後は `要件 -> 候補を削る -> 最小で満たす` で解く

## 使い方

- 最初の 30 秒で `読む前クイズ` に口頭で答える
- 表を読むときは、`左から右に読む` だけでなく、`要件から逆引きできるか` も確認する
- 読み終えたら、`双方向リコール` と `直前チェック` だけを見て内容を再現する
- 間違えた比較だけを、翌日、3日後、7日後に言い直す

## 読む前クイズ

- `最新知識を使いたい` と `答え方を変えたい` は、どの観点で切り分けるか
- `固定手順` と `推論時の自律判断` は、どの観点で切り分けるか
- `実行時に止める` と `事前に測る` は、どの観点で切り分けるか

## 比較問題の解き方

### まず比較軸を1つに絞る

- `知識を足す話か`
- `振る舞いを変える話か`
- `実行まで進める話か`
- `安全に止める話か`
- `運用しやすくする話か`

要点:

- 比較で迷うときは、候補を同じ土俵で比べようとしていることが多い
- 例えば `RAG` と `Guardrails` は競合ではなく、片方は知識補強、片方は安全制御で役割が違う
- まず `何を変える機能か` をそろえると、一気に候補が減る

## 1. 最頻出比較

### Bedrock と SageMaker AI

| 論点 | Bedrock | SageMaker AI |
| --- | --- | --- |
| 主役 | 既存 FM の利用 | 学習、微調整、配備、監視 |
| 向く場面 | 早く GenAI アプリを作る | 深く制御して MLOps まで回す |
| 代表論点 | Knowledge Bases、Guardrails、Agents | JumpStart、Pipelines、Model Monitor |

### RAG と fine-tuning

| 論点 | RAG | fine-tuning |
| --- | --- | --- |
| 何を変えるか | 推論時の外部知識 | モデルの振る舞い |
| 向く場面 | 最新知識、社内文書参照 | 出力形式、口調、分類 |
| 第一候補 | Knowledge Bases | Bedrock customization / SageMaker |

### Amazon Q Business と Knowledge Bases と Agents

| 論点 | Amazon Q Business | Knowledge Bases | Agents |
| --- | --- | --- | --- |
| 主役 | マネージド業務アシスタント | RAG 基盤 | ツール実行型エージェント |
| 向く場面 | 社員向け検索、要約、引用、権限連動 | 自作アプリに検索機能を組み込む | 予約、申請、更新など実行まで進める |
| 第一候補 | web experience を早く出したい | 独自 UI や独自 API を作りたい | 外部 API を呼んで完了まで進めたい |

### fine-tuning と continued pre-training

| 論点 | fine-tuning | continued pre-training |
| --- | --- | --- |
| データ | 入力と正解出力のペア | ラベルなし文書 |
| 目的 | 答え方を整える | ドメイン知識を厚くする |

### Distillation と reinforcement fine-tuning

| 論点 | Distillation | Reinforcement fine-tuning |
| --- | --- | --- |
| 何を与えるか | 教師モデルの応答 | reward function |
| 目的 | 小型モデルへ能力移植 | 採点基準に沿って磨く |

### Agents と Flows と Step Functions

| 論点 | Agents | Flows | Step Functions |
| --- | --- | --- | --- |
| 特徴 | 状況に応じて判断 | 固定フロー | 制御性が高いオーケストレーション |
| 向く場面 | 対話型業務処理 | 定型 GenAI 処理 | 再試行、承認、分岐が必要 |

### Guardrails と Evaluations と Clarify

| 論点 | Guardrails | Evaluations | Clarify |
| --- | --- | --- | --- |
| 何をするか | 実行時に防ぐ | 事前に測る | 公平性、説明可能性 |
| 位置づけ | Bedrock の安全制御 | Bedrock の評価 | SageMaker の Responsible AI |

### Model Monitor と CloudWatch

| 論点 | Model Monitor | CloudWatch |
| --- | --- | --- |
| 主対象 | モデル品質、ドリフト | 運用メトリクス、ログ、アラート |
| 向く場面 | ML 本番監視 | 全体運用監視 |

### prompt caching と semantic caching

| 論点 | prompt caching | semantic caching |
| --- | --- | --- |
| 何を再利用するか | 共通入力 | 類似質問への結果 |
| 向く場面 | 長い固定 prompt | FAQ 的な大量類似問い合わせ |

### provider switching と model routing と graceful degradation

| 論点 | provider switching | model routing | graceful degradation |
| --- | --- | --- | --- |
| 主目的 | 差し替えやすさ | 平常時の最適化 | 障害時も止めない |
| 向く場面 | ベンダー変更や抽象化 | 問い合わせ難度で振り分け | 障害、スロットル、容量不足 |

### Guardrails と human review

| 論点 | Guardrails | human review |
| --- | --- | --- |
| 役割 | 実行時に機械で止める | 高リスク時に人が最終確認する |
| 向く場面 | PII、禁止話題、grounding | 返金、削除、支払、権限変更 |

### Real-time、Serverless、Async、Batch

| 方式 | 向く場面 |
| --- | --- |
| `Real-time` | 即時応答 |
| `Serverless` | 断続利用 |
| `Async` | 重くて待てる |
| `Batch` | 夜間など一括処理 |

## 1.1 双方向リコール

- `早く GenAI アプリを作りたい` と聞かれたら、`Bedrock` を即答できるか
- `学習、微調整、配備、監視まで深く制御したい` と聞かれたら、`SageMaker AI` を即答できるか
- `最新知識を参照したい` と聞かれたら、`RAG` を即答できるか
- `出力形式や口調を変えたい` と聞かれたら、`fine-tuning` を即答できるか
- `社員向けの権限連動アシスタントを早く出したい` と聞かれたら、`Amazon Q Business` を即答できるか
- `固定フローを組みたい` と聞かれたら、`Flows` や `Step Functions` を即答できるか
- `推論時にツール選択まで任せたい` と聞かれたら、`Agents` を即答できるか
- `本番の入出力を止めたい` と聞かれたら、`Guardrails` を即答できるか
- `本番前に比較評価したい` と聞かれたら、`Evaluations` を即答できるか
- `断続利用の推論コストを抑えたい` と聞かれたら、`Serverless Inference` を即答できるか

## 2. 問題文のキーワードから切る

### `最新情報を参照して答えたい`

- 第一候補は `RAG`
- fine-tuning を先に選ばない

### `社員向けに権限付きの社内アシスタントを早く出したい`

- `Amazon Q Business`
- 引用、コネクタ、web experience、権限制御をまとめて使える

### `出力形式を JSON に固定したい`

- `structured outputs`
- `JSON Schema`

### `不適切出力や PII を止めたい`

- `Guardrails`
- 必要なら `Comprehend` や `Macie`

### `返金や更新処理まで自動化したい`

- `Agents` + ツール連携
- 高リスクなら human review

### `レイテンシを下げたい`

- ストリーミング
- キャッシュ
- 軽量モデル
- 入出力削減

### `コストを下げたい`

- token efficiency
- モデルルーティング
- キャッシュ
- Batch

### `説明責任が必要`

- source attribution
- Model Cards
- Lineage
- CloudTrail / decision logs

### `止めずに品質を落としてでも返したい`

- `graceful degradation`
- fallback model
- retrieval-only response

### `将来プロバイダを差し替えたい`

- 抽象化レイヤ
- `AppConfig`
- `provider switching`

### `高リスク操作なので人に確認させたい`

- `A2I`
- `Step Functions`
- human review

## 3. まず外すべき誤答

- 最新知識問題で fine-tuning を選ぶ
- 固定フローで Agents を選ぶ
- 実行時安全制御で Evaluations を選ぶ
- 公平性評価で CloudWatch だけを選ぶ
- 閉域要件で単なる暗号化だけを選ぶ
- コスト問題で最初から上位モデル固定を選ぶ
- 障害対策で `model routing` だけを選ぶ
- 高リスク業務で `Guardrails` だけに頼る

## 4. 解答パターン

### パターン1: 最小で満たす

- 余計なサービスを足しすぎない
- 問われた要件だけを最も自然に満たす構成を選ぶ

### パターン2: 生成 AI 固有の原因を先に疑う

- 回答品質低下 -> prompt、RAG、context overflow
- コスト増加 -> token、キャッシュ、モデル選択
- 安全性 -> Guardrails、prompt injection、多層防御

### パターン3: 運用まで見る

- 作れるかだけでなく、監査、アラート、ロールバックを持つか

### パターン4: 主役と補助を分ける

- `Knowledge Bases` が主役で、`Guardrails` は補助
- `Agents` が主役で、`A2I` は高リスク時の補助
- `Model evaluation` が主役で、`CloudWatch` は運用監視の補助

補足:

- 誤答は、補助機能を主役と取り違えたものが多い
- `主役の課題を直接解くか` を見ると、かなり消去しやすい

## 5. 直前チェック

- Bedrock と SageMaker AI の境目を説明できるか
- RAG と fine-tuning の使い分けを 1 文で言えるか
- Guardrails と Evaluations の違いを 1 文で言えるか
- provider switching、model routing、graceful degradation の違いを 1 文で言えるか
- Agent と Flow の違いを 1 文で言えるか
- prompt caching と semantic caching の違いを 1 文で言えるか
- Real-time、Serverless、Async、Batch を即断できるか
- PII、監査、閉域接続の答えを即断できるか
- A/B テスト、カナリア、ロールバックの役割を言えるか
- `Guardrails` と human review を併用すべき場面を言えるか

## 復習の回し方

- 初回は `読む前クイズ` -> 表 -> `双方向リコール` の順で確認する
- 翌日は表を見ずに、`双方向リコール` と `直前チェック` だけに答える
- 3日後は `まず外すべき誤答` を見て、自分が引っかかりやすい誤答を 3 つ言う
- 7日後は全文を読み返さず、`要件を言われたら第一候補を返す` 練習だけに絞る

## Active Recall

- 最新知識を使わせたいときの第一候補は何か
- 出力形式を固定したいときの第一候補は何か
- 固定手順のワークフローで第一候補は何か
- 公平性や説明可能性の第一候補は何か
- 閉域接続の第一候補は何か
- 断続利用の推論方式は何か
- 障害時も止めたくないときに見る設計は何か
- 高リスク操作に人を入れたいときの第一候補は何か

### 答えと解説

1. `RAG` です。外部知識を推論時に与える方が自然です。
2. `structured outputs` と `JSON Schema` です。自由文より安全です。
3. `Flows` または `Step Functions` です。Agent は自由度が高すぎることがあります。
4. `Clarify`、`Model Cards`、説明ログです。単なる性能監視では足りません。
5. `VPC endpoints / PrivateLink` です。経路制御の論点です。
6. `Serverless Inference` です。アイドル時間が多い要件に向きます。
7. `graceful degradation` です。小型モデル、キャッシュ、検索のみ応答などに段階的に落とします。
8. `A2I` や `Step Functions` を使った human review です。自動化しすぎないのが重要です。
