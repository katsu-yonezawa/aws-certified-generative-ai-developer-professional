# 頻出比較と解答パターン

## まず結論

- AIP-C01 は `比較問題` が多い
- サービス名を丸暗記するより、`何を変える機能か`、`何を見る機能か` を見分ける
- 最後は `要件 -> 候補を削る -> 最小で満たす` で解く

## 使い方

- 最初の 30 秒で `読む前クイズ` に口頭で答える
- 表は `左から右に読む` だけでなく `要件から逆引きできるか` も確認する
- 間違えた比較だけを、翌日、3日後、7日後に言い直す

## 読む前クイズ

- `最新知識を使いたい` と `答え方を変えたい` は、どの観点で切り分けるか
- `固定手順` と `推論時の自律判断` は、どの観点で切り分けるか
- `実行時に止める` と `事前に測る` は、どの観点で切り分けるか

## 比較問題の解き方

### まず比較軸を 1 つに絞る

- `知識を足す話か`
- `振る舞いを変える話か`
- `実行まで進める話か`
- `安全に止める話か`
- `運用しやすくする話か`
- `何を記録する話か`

要点:

- 比較で迷うときは、候補を同じ土俵で比べていないことが多い
- まず `何を変える機能か`、`何を見る機能か` をそろえると候補が減る

## 1. 最頻出比較

### Bedrock と SageMaker AI

| 論点 | Bedrock | SageMaker AI |
| --- | --- | --- |
| 主役 | 既存 FM の利用 | 学習、微調整、配備、監視 |
| 向く場面 | 早く GenAI アプリを作る | 深く制御して MLOps まで回す |
| 代表論点 | Knowledge Bases、Guardrails、Agents | JumpStart、Pipelines、Model Monitor |

### RAG と Fine-tuning

| 論点 | RAG | Fine-tuning |
| --- | --- | --- |
| 何を変えるか | 推論時の外部知識 | モデルの振る舞い |
| 向く場面 | 最新知識、社内文書参照 | 出力形式、口調、分類 |
| 第一候補 | Knowledge Bases | Bedrock customization / SageMaker |

### Amazon Q Business と Knowledge Bases と Agents

| 論点 | Amazon Q Business | Knowledge Bases | Agents |
| --- | --- | --- | --- |
| 主役 | マネージド業務アシスタント | RAG 基盤 | ツール実行型エージェント |
| 向く場面 | 社員向け検索、要約、引用、権限連動 | 自作アプリに検索機能を組み込む | 予約、申請、更新など実行まで進める |

### Agents と Flows と Step Functions

| 論点 | Agents | Flows | Step Functions |
| --- | --- | --- | --- |
| 特徴 | 状況に応じて判断 | 固定フロー | 制御性が高いオーケストレーション |
| 向く場面 | 対話型業務処理 | 定型 GenAI 処理 | 再試行、承認、分岐が必要 |

### Prompt management と Prompt caching

| 論点 | Prompt management | Prompt caching |
| --- | --- | --- |
| 主目的 | プロンプト資産化 | 入力再利用による最適化 |
| 向く場面 | 変数化、比較、版管理 | 長い共通 prefix を繰り返す |

### OpenSearch と S3 Vectors

| 論点 | OpenSearch | S3 Vectors |
| --- | --- | --- |
| 主役 | 検索機能の豊富さ | 低コストで軽い運用 |
| 向く場面 | ハイブリッド検索、集計、高度な検索 | 大規模保存、低頻度検索、シンプル運用 |

### Guardrails と Evaluations と Clarify

| 論点 | Guardrails | Evaluations | Clarify |
| --- | --- | --- | --- |
| 何をするか | 実行時に防ぐ | 事前に測る | 公平性、説明可能性 |
| 位置づけ | Bedrock の安全制御 | Bedrock の評価 | SageMaker の Responsible AI |

### CloudTrail と model invocation logging と CloudWatch

| 論点 | CloudTrail | model invocation logging | CloudWatch |
| --- | --- | --- | --- |
| 何を見るか | API 操作 | 推論入出力 | 運用メトリクス |
| 向く場面 | 誰が設定を変えたか追いたい | 入出力内容を監査したい | レイテンシ、エラー、スロットルを見たい |

### Trace と CloudTrail

| 論点 | Trace | CloudTrail |
| --- | --- | --- |
| 何を見るか | Agent や Flow の内部手順 | AWS API 操作 |
| 向く場面 | どのツールを選んだか追いたい | 誰が変更したか追いたい |

### Prompt caching と semantic caching

| 論点 | Prompt caching | semantic caching |
| --- | --- | --- |
| 何を再利用するか | 共通入力 | 類似質問への結果 |
| 向く場面 | 長い固定 prompt | FAQ 的な大量類似問い合わせ |

### Intelligent prompt routing と graceful degradation

| 論点 | Intelligent prompt routing | graceful degradation |
| --- | --- | --- |
| 主目的 | 平常時の品質とコスト最適化 | 障害時も止めない |
| 向く場面 | 難問だけ上位モデルへ回したい | スロットルや障害時に縮退運転したい |

### provider switching と model routing と graceful degradation

| 論点 | provider switching | model routing | graceful degradation |
| --- | --- | --- | --- |
| 主目的 | 差し替えやすさ | 平常時の最適化 | 障害時も止めない |

### Guardrails と human review

| 論点 | Guardrails | human review |
| --- | --- | --- |
| 役割 | 実行時に機械で止める | 高リスク時に人が最終確認する |
| 向く場面 | PII、禁止話題、grounding | 返金、削除、支払、権限変更 |

## 2. 問題文のキーワードから切る

### `最新情報を参照して答えたい`

- 第一候補は `RAG`
- Fine-tuning を先に選ばない

### `社員向けに権限付きの社内アシスタントを早く出したい`

- `Amazon Q Business`
- 引用、コネクタ、web experience、権限制御をまとめて使える

### `出力形式を JSON に固定したい`

- `structured outputs`
- `JSON Schema`

### `不適切出力や PII を止めたい`

- `Guardrails`
- 必要なら `Comprehend` や `Macie`

### `返金や更新処理まで自動化したい`

- `Agents` + ツール連携
- 高リスクなら human review

### `レイテンシを下げたい`

- 入出力削減
- `Prompt caching`
- 軽量モデル
- ストリーミング

### `説明責任が必要`

- source attribution
- `CloudTrail`
- `model invocation logging`
- `Model Cards`

### `構造化 / 半構造化 / 非構造化データをどう扱うか迷う`

- 構造化はそのまま問い合わせるか文書化するかを先に切る
- 非構造化は `Textract`、`Transcribe`、`Bedrock Data Automation` などの前処理を疑う

### `検索機能が豊富なベクトルストアが必要`

- `OpenSearch`

### `低コストで大規模なベクトル保存をしたい`

- `S3 Vectors`

## 3. まず外すべき誤答

- 最新知識問題で Fine-tuning を選ぶ
- 固定フローで Agents を選ぶ
- 実行時安全制御で Evaluations を選ぶ
- 監査要件で `CloudWatch` だけを選ぶ
- `Trace` だけで監査要件を満たしたつもりになる
- `Action group = OpenAPI + Lambda 固定` と考える
- コスト問題で最初から上位モデル固定を選ぶ

## 4. 解答パターン

### パターン1: 最小で満たす

- 余計なサービスを足しすぎない
- 問われた要件だけを最も自然に満たす構成を選ぶ

### パターン2: 主役と補助を分ける

- `Knowledge Bases` が主役で、`Guardrails` は補助
- `Agents` が主役で、human review は高リスク時の補助
- `Evaluations` が主役で、`CloudWatch` は運用監視

### パターン3: 何を記録するかを分ける

- API 操作 -> `CloudTrail`
- 推論入出力 -> `model invocation logging`
- 内部手順 -> `Trace`
- 性能監視 -> `CloudWatch`

## 5. 直前チェック

- Bedrock と SageMaker AI の境目を説明できるか
- RAG と Fine-tuning の使い分けを 1 文で言えるか
- Agents と Flows と Step Functions の違いを 1 文で言えるか
- Prompt management と Prompt caching の違いを 1 文で言えるか
- OpenSearch と S3 Vectors の違いを 1 文で言えるか
- Guardrails と Evaluations の違いを 1 文で言えるか
- `CloudTrail`、`model invocation logging`、`CloudWatch` の違いを即答できるか
- `Trace` と `CloudTrail` の違いを即答できるか
- `Guardrails` と human review を併用すべき場面を言えるか

## Active Recall

- `最新知識を使いたい` と聞かれたら何を即答するか
- `固定手順の GenAI ワークフロー` と聞かれたら何を即答するか
- `推論時に状況依存でツール選択したい` と聞かれたら何を即答するか
- `長い共通 prefix を何度も送る` と聞かれたら何を即答するか
- `API 操作監査` と `推論入出力監査` は何が違うか
- `Trace` は何を見る仕組みか
- `検索機能の豊富さ` と `軽い運用` はどの比較か

### 答えと解説

1. `RAG` です。最新知識や社内文書を推論時に引く問題なので、Fine-tuning より自然です。
2. `Flows` です。分岐やノードを含めても、手順が決まっているならこちらが主役です。
3. `Agents` です。会話しながらツール選択まで進むからです。
4. `Prompt caching` です。共通 prefix の入力再利用による最適化です。
5. API 操作監査は `CloudTrail`、推論入出力監査は `model invocation logging` です。残す対象が違います。
6. Agent や Flow の内部手順です。どのツールを選び、どの順で進んだかを追います。
7. `OpenSearch` と `S3 Vectors` の比較です。前者は検索機能、後者は低コストで軽い運用が軸です。
