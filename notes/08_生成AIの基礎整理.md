# 生成AIの基礎整理

## 位置づけ

このノートは試験の主戦場ではありません。  
ただし、`RAG と Fine-tuning の切り分け`、`推論パラメータの意味`、`何を先に直すべきか` を安定して判断する土台として有効です。

要点:

- 生成 AI は `文脈に条件づけられた次トークン予測` を繰り返して出力する
- 実務では `言い回し` より `何を渡すか` と `どう制約するか` と `どう評価するか` が重要
- 最新知識や社内知識は `RAG`、振る舞いや形式の固定は `Fine-tuning`

## 理解の土台

### 生成 AI をどう見るか

- モデルは `知っていることを完璧に再生する装置` ではない
- 与えられた文脈から、もっともらしい次トークン列を作る装置
- だから `何を入れるか` と `どう制約するか` が品質を大きく左右する

### 基盤モデル、ベースモデル、推論

- 広いデータで事前学習された大規模モデルが基盤モデル
- Bedrock では、提供される元のモデルをベースモデルとして扱う
- そのモデルを呼び出して応答を得ることが推論

要点:

- 試験では、いきなり学習へ飛ばず、まず `推論時の工夫 -> RAG -> カスタマイズ` の順で考える

## 1. まず押さえる仕組み

### トークン

- モデルは文章をそのまま理解するのではなく、まず `token` の列として扱う
- token 数はそのまま `コスト`、`レイテンシ`、`context window` 圧迫に効く
- 長い入力も長い出力も、どちらも運用コストに効く

### 次トークン予測

- LLM は、直前までの文脈から `次に出すトークン` を確率的に選ぶ
- そのため、同じ問いでも毎回まったく同じ結果になるとは限らない
- 自然な文章を返せても、内容の正確さまでは自動で保証されない

### 埋め込み

- テキストや画像を数値ベクトルへ変換して近さを比較しやすくする
- `Knowledge Bases`、`ベクトル検索`、`類似検索` の前提
- 埋め込みは `検索用の表現` であり、生成そのものとは役割が違う

### 文脈

- 出力は `system 指示`、`会話履歴`、`検索結果`、`ツール結果`、`出力形式指定` に強く依存する
- 同じモデルでも、渡す文脈が変われば品質は大きく変わる
- 情報が足りなくても多すぎても不安定になる

要点:

- `全部入れる` より `必要なものだけ渡す` 方が強い

## 2. プロンプトはどう考えるか

### いま重要なこと

- 目的を明示する
- 制約を明示する
- 出力形式を固定する
- 不明な場合の扱いを指定する
- 必要な資料だけを渡す

### プロンプト改善より先に見ること

1. 成功条件が定義されているか
2. 必要な文脈を渡せているか
3. 出力形式を固定しているか
4. 評価方法があるか
5. それでも不足するなら prompt やモデルを見直す

### なぜ prompt だけでは限界があるか

- prompt は `指示の明確化` には効く
- しかし `最新知識の追加`、`外部処理の実行`、`形式保証` は他の仕組みの方が強い
- そのため実務では、prompt を磨く前に `RAG`、`tool use`、`structured outputs` を検討する

## 3. 推論パラメータの見方

### temperature

- 低いほど上位候補に寄りやすく、出力が安定しやすい
- 高いほど候補が広がり、多様性が増えやすい
- `hallucination を直接なくす機能` ではない

### top-p / top-k

- 候補集合をどこまで広げるかを調整する
- `ランダムさの制御` に関わる
- 正確性の問題は、まず `文脈不足`、`検索不足`、`モデル選択不一致` を疑う

### max tokens

- 出力長の上限
- 冗長な回答を防ぎ、コストとレイテンシを抑える基本設定

試験向けの整理:

- `安定した短い業務応答` -> temperature を低めにし、出力形式を固定する
- `創造的な案出し` -> 多様性パラメータの調整余地がある
- `事実性が足りない` -> まず RAG、参照データ、評価を見直す

## 4. RAG とカスタマイズ

### RAG が第一候補になる場面

- 最新知識を使いたい
- 社内文書を根拠付きで使いたい
- 文書更新のたびに再学習したくない

### Fine-tuning が第一候補になる場面

- 出力形式や口調をそろえたい
- 分類の癖を安定させたい
- 定型応答を強く安定させたい

要点:

- `知識を足したい` と `振る舞いを変えたい` を混同しない

## 5. よくある誤解

### 低い temperature なら正しくなる

- 誤りです
- 低温では答えが安定しやすいだけで、安定して間違うこともある

### 長い context を全部入れれば解決する

- 誤りです
- ノイズが増え、重要情報が埋もれ、コストも上がる
- 必要な情報だけを検索して渡す方がよい

### 最新知識が欲しいなら Fine-tuning

- 多くは誤りです
- 最新文書、社内文書、頻繁更新データは `RAG` や外部データ参照が第一候補

### 出力形式のブレは prompt を長くすれば解決する

- 限界があります
- `structured outputs`、`schema 制約`、`tool use` の方が実装では強い

## 6. 試験との接続

### Domain 1

- `トークン`、`埋め込み`、`チャンク分割`、`ベクトル検索` を一つの流れで理解しやすくなる
- `最新知識は RAG、振る舞い変更は Fine-tuning` を切り分けやすくなる

### Domain 2

- `prompt だけで解くか`、`tool use を使うか`、`Agents にするか` の判断が安定する
- `structured outputs` を優先する理由が明確になる

### Domain 4

- `token efficiency`、`Prompt caching`、`max tokens` の意味がつながる
- レイテンシとコストの改善で `まず入力を減らす` 発想を持てる

### Domain 5

- 品質改善では `思いつきの prompt 修正` より `evaluation` が先だと整理できる
- 変更の比較には `同じ入力集合` と `同じ採点基準` が必要

## Active Recall

- 生成 AI の出力は何に最も強く依存するか
- `temperature` は何を調整し、何を保証しないか
- `top-p` や `top-k` は何の調整か
- 最新知識を答えさせたいとき、第一候補は何か
- 出力形式のブレを減らしたいとき、自由文の工夫だけで十分か
- prompt 改善の前に定義すべきものは何か

### 答えと解説

1. 文脈です。system 指示、履歴、検索結果、ツール結果、出力形式指定が効きます。
2. 出力の多様性やばらつきです。事実性や正しさは保証しません。
3. 候補集合の広さです。ランダムさの調整に関わります。
4. 多くのケースでは `RAG` や外部データ参照です。更新知識を重みに焼き込むより自然です。
5. 十分ではありません。`structured outputs` や schema 制約の方が安定します。
6. 成功条件と評価方法です。ここが曖昧だと改善比較ができません。
