# 実戦問題集

## まず使い方

- 1周目は解説を見ず、各問を 60 秒前後で解く
- 2周目は正答だけでなく、誤答がなぜ違うかを 1 文で言う
- 3周目は各問の `1文で言うと` だけを見て口頭再現する
- 間違えた問題だけを翌日、3日後、7日後に解き直す

## 解く前クイズ

- `最新文書を使いたい` と `答え方を調整したい` は、どの観点で切り分けるか
- `固定フロー` と `推論時のツール判断` は、どの観点で切り分けるか
- `本番で止める` と `本番前に比較する` は、どの観点で切り分けるか

## 解説の読み方

- 正答のサービス名だけで終わらず、`何を解決する機能か` を言い換える
- 誤答は `なぜ今その選択肢ではないか` を 1 文で残す
- 同じ誤りが続くなら、`知識不足`、`安全性の見落とし`、`運用要件の見落とし` のどれかに分類する

要点:

- 問題集は正解数を競うより、`誤答パターンを減らす` ために使う
- AIP-C01 は似た比較が繰り返し出るため、誤答理由を言語化すると伸びやすい

## 誤答ログ

| 問番 | 自分の誤答 | 正答 | 取り違えた論点 | 次回の 1 文 |
| --- | --- | --- | --- | --- |
| 例: 問4 | Agents | Flows | 固定フローと自律判断を混同 | 固定手順なら Flows、推論時判断なら Agents |

## 解答と解説

### 問1

答え: B

- 本質は `知識を推論時に差し込むか、学習でモデル内部に持たせるか` です。文書が日々変わるなら、モデルを再学習するより RAG の方が自然です。Knowledge Bases はデータソース取り込み、検索、RetrieveAndGenerate の流れを支えます。
- A は出力の振る舞い調整には有効でも、最新文書の追従には不向きです。
- C はドメイン知識の厚みには効きますが、頻繁に変わる運用文書を即反映する設計ではありません。
- D はモデル能力を上げても、外部知識を参照しない限り `最新性` の論点は解けません。
- 1文で言うと: `最新知識が論点なら、まず RAG を考える`

### 問2

答え: A

- 本質は `既存 FM を早く安全に使うのか、自前で学習や MLOps まで深く制御するのか` です。複数 FM への単一 API アクセス、Guardrails、Knowledge Bases という要件は Bedrock と自然に整合します。
- B はモデル構築、学習、配備、監視まで深く制御したいときの主役です。
- C と D はこの要件の中核ではありません。
- 1文で言うと: `早く GenAI アプリを作るなら Bedrock、深く ML を回すなら SageMaker AI`

### 問3

答え: A

- 本質は `再利用可能な prompt を変数付きで管理し、variant を比較し、版管理する` ことです。Prompt management はそのための機能です。
- B は安全制御であり、prompt の版管理や比較の主役ではありません。
- C は観測用途であり、prompt の再利用管理ではありません。
- D は API 監査ログであり、prompt 開発の主機能ではありません。
- 1文で言うと: `prompt を資産として管理するなら Prompt management`

### 問4

答え: B

- 本質は `固定フローか、自律判断か` です。定義済みの手順で prompt ノードや条件分岐を組むなら Flows が自然です。
- A はモデルが状況に応じてツール選択や手順判断を行う用途に向きます。
- C はワークフロー制御ではありません。
- D は安全制御であり、処理手順の組み立てではありません。
- 1文で言うと: `決めた順に流すなら Flows`

### 問5

答え: B

- 本質は `ユーザー意図に応じてどのツールを使うかを推論時に決める` ことです。Agents はモデルにツール利用を伴う実行判断をさせる用途に向きます。
- A は固定手順なら有効ですが、ここでは毎回の判断が必要です。
- C はレイテンシ・コスト最適化です。
- D は一括非同期処理であり、対話的なツール利用ではありません。
- 1文で言うと: `ツール選択をモデルに任せるなら Agents`

### 問6

答え: A

- 本質は `自由文で頑張るのでなく、出力形式自体を制約する` ことです。Structured outputs は JSON schema や strict tool use により、機械可読な出力を安定化します。
- B はランダム性を下げても schema 準拠を保証しません。
- C はログであって、出力制約ではありません。
- D も形式保証には直結しません。
- 1文で言うと: `JSON を壊したくないなら、prompt ではなく schema で縛る`

### 問7

答え: B

- 本質は `実行時に危険な入出力を制御する` ことです。Guardrails は denied topics、content filters、sensitive information filters などを提供します。
- A は事前評価であり、実行時ブロックではありません。
- C は prompt 管理です。
- D は ML モデル監視であり、チャット安全制御の第一候補ではありません。
- 1文で言うと: `本番の安全制御は Guardrails`

### 問8

答え: B

- 本質は `アイドルを抱えるか、必要時だけ実行するか` です。断続利用でコスト最適化したいなら Serverless Inference が自然です。
- A は常時エンドポイント前提で、即時性は高い一方アイドルコストが発生しやすいです。
- C は一括処理向けです。
- D は大きな payload や長時間処理向けで、即時応答の第一候補ではありません。
- 1文で言うと: `断続利用なら Serverless Inference`

### 問9

答え: A

- 本質は `長くて繰り返す共通 prefix を毎回再計算しない` ことです。Prompt caching はレイテンシと入力 token コストを下げるための機能です。
- B は振る舞い変更のための学習であり、共通文脈の再利用ではありません。
- C は監査です。
- D はワークフローです。
- 1文で言うと: `長い共通文脈を何度も送るなら prompt caching`

### 問10

答え: A

- 本質は `同じデータセットで候補を比較し、評価方法を揃える` ことです。Model evaluation は自動評価や human-based evaluation を使い分けられます。
- B は運用監視であり、評価ジョブの主役ではありません。
- C は安全制御です。
- D は推論最適化です。
- 1文で言うと: `候補比較は Bedrock の評価基盤で行う`

### 問11

答え: B

- 本質は `推論そのものの入力と出力を残す` ことです。Model invocation logging はリクエスト、レスポンス、メタデータを CloudWatch Logs や S3 に送れます。
- A は AWS API コールの監査には有効ですが、モデル入出力本文の継続保存とは別です。
- C は分散トレースです。
- D は特定ワークフローの履歴であり、モデル呼び出し本文の記録用途とは違います。
- 1文で言うと: `モデルの中身を追うなら invocation logging`

### 問12

答え: A

- 本質は `取れた候補の順位を、質問との関連性でもう一度整える` ことです。Reranker は取得済み文書を並べ替え、より関連性の高い結果に寄せます。
- B は共通 prompt の再利用であり、検索順位改善ではありません。
- C は処理方式の違いです。
- D はモデル知識の追加であり、検索順位の即効策ではありません。
- 1文で言うと: `拾えているが順位が悪いなら rerank`

### 問13

答え: A

- 本質は `単一リージョン固定でなく、推論リソースへの入口を抽象化する` ことです。Inference profiles は cross-Region inference によるスループット拡大や可用性向上に使えます。
- B は安全制御です。
- C は prompt 管理です。
- D は評価です。
- 1文で言うと: `地理圏内で推論先を広げるなら inference profiles`

### 問14

答え: A

- 本質は `性能だけでなく、説明可能性とバイアスも見る` ことです。SageMaker Clarify は説明可能性やバイアス検出の論点で中心になります。
- B は推論最適化です。
- C はワークフローです。
- D は配信サービスであり、この論点には無関係です。
- 1文で言うと: `説明可能性とバイアスは Clarify`

### 問15

答え: A

- 本質は `特定モデルの呼び出し詳細をアプリ本体へ埋め込まない` ことです。抽象化レイヤを置き、設定を AppConfig のように外出しすると provider switching をしやすくなります。
- B は切り替えのたびにアプリ改修が増えます。
- C は将来の差し替え耐性を下げます。
- D は推論最適化であり、差し替え容易性の論点ではありません。
- 1文で言うと: `将来差し替えるなら、モデル呼び出しを抽象化する`

### 問16

答え: B

- 本質は `高リスク操作だけ human-in-the-loop にする` ことです。A2I や Step Functions を使うと、承認、差し戻し、再実行の流れを組みやすくなります。
- A は安全制御に有効でも、人の最終判断の代替ではありません。
- C は出力のばらつき調整です。
- D は prompt の資産管理です。
- 1文で言うと: `高リスク操作は human review を足す`

### 問17

答え: A

- 本質は `誰がどのデータへ触れるかを粒度細かく統制する` ことです。Lake Formation はデータ湖の権限制御で中心になります。
- B は推論最適化です。
- C はワークフローです。
- D はログ保存であり、アクセス制御ではありません。
- 1文で言うと: `データ湖の細粒度統制は Lake Formation`

### 問18

答え: A

- 本質は `モデル全体ではなく差分だけを学習する` ことです。LoRA や adapters は parameter-efficient adaptation の代表で、コストと保存量を抑えやすいです。
- B は推論時知識参照です。
- C は入力再利用です。
- D は監査です。
- 1文で言うと: `安く速く適応するなら LoRA / adapters`

### 問19

答え: A

- 本質は `検索前のデータ品質を整える` ことです。欠損、重複、メタデータ不整合が多いなら、まず Glue Data Quality のような品質検証を疑います。
- B は実行時安全制御です。
- C は性能保証です。
- D はリージョン跨ぎの推論です。
- 1文で言うと: `埋め込み前の汚れはデータ品質で直す`

### 問20

答え: A

- 本質は `傾向把握` と `異常検知` を分けることです。Cost Explorer は可視化、Cost Anomaly Detection は急増検知に向きます。
- B は監査とトレースです。
- C と D はコスト管理の主役ではありません。
- 1文で言うと: `普段は Cost Explorer、急増検知は anomaly detection`

### 問21

答え: A

- 本質は `代表フローを定期実行して体験ベースで監視する` ことです。CloudWatch Synthetics は疎通や主要操作の継続監視に向きます。
- B は API 操作監査です。
- C は推論最適化です。
- D は説明可能性とバイアスです。
- 1文で言うと: `代表フローの定期監視は Synthetics`

### 問22

答え: A

- 本質は `更新をしきい値で止めるか、通すか` を決めることです。quality gate と synthetic user workflow を置くと、主観ではなく基準で更新判断できます。
- B は安定化の一部でしかありません。
- C は安全制御であり、更新判定の主機能ではありません。
- D は品質保証を置き換えません。
- 1文で言うと: `更新前に quality gate を置く`

### 問23

答え: A

- 本質は `Agent を回答文だけでなく、完了能力まで測る` ことです。AgentCore Evaluations は task completion、tool usage、multi-step reasoning の評価に向きます。
- B は最適化です。
- C は観測には効きますが、体系的評価とは違います。
- D は安全制御です。
- 1文で言うと: `Agent は完了率とツール利用まで評価する`

### 問24

答え: A

- 本質は `障害時に品質を少し落としてでも返す` ことです。graceful degradation では、小型モデル、キャッシュ、検索結果のみ応答などへ段階的に落とせます。
- B は学習手法です。
- C は由来管理です。
- D は監査です。
- 1文で言うと: `止めない設計は graceful degradation`

### 問25

答え: D

- 本質は `一時的失敗への再試行` ではなく `継続的に不調な依存先を一時遮断して全体を守る` ことです。AWS Prescriptive Guidance の circuit breaker pattern は、繰り返すタイムアウトや失敗を検知したら呼び出しを止め、即時失敗で返す設計です。
- A は一時的な障害には有効ですが、継続障害の相手を呼び続ける点は変わりません。
- B は不調な依存先への負荷と失敗数を増やしやすく、逆効果です。
- C は障害検知を遅らせ、待機によるリソース消費を増やします。
- 1文で言うと: `障害が続く相手には retry ではなく circuit breaker`

## 復習の観点

### 何で間違えたかを分類する

- `主役サービスを取り違えた`
- `安全策を補助機能として見落とした`
- `運用要件より機能名で選んだ`
- `RAG と fine-tuning のような比較軸を混同した`

### 解き直しで見ること

1. 問題文の主語は `検索`、`実行`、`安全性`、`評価` のどれだったか
2. 制約は `最新性`、`閉域`、`低コスト`、`高リスク` のどれだったか
3. 正答は `直接解く機能` だったか、補助機能だったか

要点:

- この観点で見直すと、単なる暗記ではなく `判断の癖` を修正できる
- 1 周ごとに `なぜその答えになるか` を短く口頭再現すると定着しやすい

- Bedrock と SageMaker AI の境目を 1 文で言えるか

- provider switching、model routing、graceful degradation の違いを言えるか
- retry と circuit breaker の違いを `一時的失敗への再試行` と `継続障害の遮断` で言えるか
- A2I、Lake Formation、Glue Data Quality がどの層の論点か言えるか
- quality gate、synthetic user workflow、Agent 評価を 1 文で言えるか
