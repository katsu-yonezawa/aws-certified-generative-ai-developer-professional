# 実装と統合

## まず結論

- Domain 2 は `FM をどう業務に組み込むか` の問題
- 論点は `Agent`、`ツール連携`、`API 呼び出し`、`オーケストレーション`、`既存システム統合`
- 問題文では機能名より `要件に対してどの構成が自然か` を問われる

## このドメインでまず切ること

### 1. 推論だけか、実行まで必要か

- 推論だけでよい -> `Converse`、`Responses`、`InvokeModel`
- 外部 API 実行まで必要 -> `tool use`、`Agents`
- 手順が固定 -> `Flows`、`Step Functions`

### 2. 会話か、バックエンド処理か

- 会話アプリ -> 会話履歴、ストリーミング、状態管理が重要
- バックエンド処理 -> 非同期、再試行、イベント駆動が重要

### 3. 1 回で終わるか、複数ステップか

- 単発推論 -> シンプルな API 統合
- 複数ステップ -> オーケストレーションが必要

## 構成を選ぶときの考え方

### まず `自由度` と `制御性` を比べる

- `Agents` は自由度が高いが、結果のばらつきや安全制御を強く意識する
- `Flows` は固定手順に強いが、状況依存の判断には弱い
- `Step Functions` は制御性、再試行、監査、承認に強い

要点:

- 実装問題では `高機能なもの` より `要件に対して過不足がないもの` を選ぶ
- 固定手順なのに `Agents` を選ぶ、単純な同期 API なのに `Step Functions` を選ぶ、が典型的な誤り

## 1. エージェンティック AI とツール連携

### tool use

- モデルが外部ツールを選び、引数を生成し、結果を再利用する考え方
- 予約、検索、計算、CRM 更新などに向く
- 重要なのは `どのツールを呼べるか` と `どこまで権限を与えるか`

### Agents

- 会話しながら計画し、必要なツールを呼び、完了まで進める
- Bedrock の Agents は業務 API を組み込むときの第一候補
- ただし万能ではなく、固定手順なら過剰

#### 図で見る Agents の実行ループ

![Agents for Amazon Bedrock の実行ループ](./img/bedrock-agents-runtime.png)

要点:

- `前処理プロンプト -> 会話履歴 -> オーケストレーション -> Action Group / Knowledge Base -> 観測を戻して再実行` の形で進む
- `状況に応じてツール選択する` のが Agents、`順序が固定` なら Flows と切り分ける

出典: [AWS 公式図](https://docs.aws.amazon.com/images/bedrock/latest/userguide/images/agents/agents-runtime.png)

### Flows

- ステップ順や分岐があらかじめ決まっている処理向け
- 文書要約 -> 分類 -> 承認依頼 のような固定フローに向く

### 覚え方

- `対話しながら判断` -> `Agents`
- `決まった順序で処理` -> `Flows`
- `モデルに API を叩かせる` -> `tool use`

## 2. Agent 周辺の頻出論点

### 状態管理とメモリ

- セッションまたぎで文脈が必要か
- ユーザーごとに会話履歴を分離するか
- 長期記憶を持たせるか

試験では:

- 状態を持たせるなら保存先と権限境界も見る
- 会話履歴を持つほど、PII、コスト、コンテキスト圧迫が増える

補足:

- `全部の履歴を残す` は自然ではなく、`何を残すと次回判断に効くか` を選ぶことが重要
- 長期記憶が必要でも、会話全文ではなく `要約メモリ` や `業務状態` だけ残す方が安定しやすい

### ツールの安全制御

- IAM で最小権限
- Lambda で入力検証
- Step Functions でタイムアウト、リトライ、分岐
- サーキットブレーカーで障害拡大を防ぐ

### 人手承認

- 高リスクな更新処理は human-in-the-loop を疑う
- 返金、支払、設定変更、削除系は自動化しすぎない

### human augmentation

- `Amazon Augmented AI`
  人手レビュー工程を標準化したいときの選択肢
- `Step Functions`
  承認、差し戻し、再実行を含む review フローを組みやすい

覚え方:

- `人の判断を挟みたい` -> `A2I` や `Step Functions`
- `全部を人に戻す` ではなく、`高リスクだけ人に渡す`

## 3. Strands Agents、AWS Agent Squad、MCP、Nova Act、AgentCore、A2I

### ざっくり位置づけ

- `Strands Agents`
  AWS のオープンソース SDK。複雑なエージェント実装を組みやすい
- `AWS Agent Squad`
  複数エージェント協調の例として試験ガイドに出る
- `MCP`
  Agent とツールの接続を標準化する考え方
- `Amazon Nova Act`
  ブラウザ UI 操作を含むエージェント自動化
- `Amazon Bedrock AgentCore`
  エンタープライズ向けのエージェント実装基盤
- `Amazon Augmented AI`
  人手レビューを組み込むためのサービス

### 試験での扱い

- これらは `Agent をどう実装するか` の選択肢
- 中心論点は名前暗記ではなく `複数ツール / 複数エージェント / 外部ツール標準接続 / UI 自動化 / 人手補完 / 実行基盤` を見分けること

## 4. FM API 統合

### 主な呼び出しの切り分け

| 論点 | 第一候補 |
| --- | --- |
| 新規会話アプリ | `Responses API` |
| 複数モデルを統一形式で扱う | `Converse API` |
| OpenAI 互換コードの移植 | `Chat Completions API` |
| 埋め込み、画像生成、モデル固有パラメータ | `InvokeModel` |

### ストリーミング

- レイテンシ体感を下げたいときに有効
- 長文応答や UI 体験改善に向く
- ただし後段システムが JSON 完全体を必要とするなら相性に注意

補足:

- リアルタイム UI では `WebSockets` や `server-sent events` が出やすい
- API Gateway のストリーミングや chunked transfer の文脈と結び付ける

### リトライと冪等性

- 一時的エラーは再試行
- 更新系は重複実行を防ぐ
- タイムアウトと再試行回数は分けて設計する

### graceful degradation と model routing

- 障害時は `高性能モデル -> 小型モデル -> 検索のみ応答` のように段階的に落とす
- `graceful degradation` は止めないための設計
- `model routing` は問い合わせごとに最適なモデルへ振り分ける設計
- `Step Functions` やアプリコードで静的 / 動的ルーティングを実装できる

## 5. 企業システム統合

### よくある構成

- `API Gateway + Lambda + Bedrock`
  最小構成。同期 API に向く
- `Step Functions + Lambda + Bedrock`
  複数ステップ、承認、分岐、再試行に向く
- `EventBridge + SQS/SNS + Lambda`
  疎結合、非同期、イベント駆動に向く
- `App Runner / ECS / EKS`
  コンテナアプリとして推論周辺処理を運用したいとき
- `GenAI gateway`
  複数モデル、監査、ポリシー、レート制御を集約したいとき

### どう選ぶか

- 即時応答 -> `API Gateway`
- 長いワークフロー -> `Step Functions`
- 非同期連携、疎結合 -> `EventBridge`、`SQS`
- 常駐アプリや独自ランタイム -> `App Runner`、`ECS`、`EKS`
- 複数チームで統制したい -> `gateway` パターン

### 統合方式を誤らないための見方

- `同期で返す必要があるか`
- `失敗時にあとで再実行できればよいか`
- `1つの障害が全体へ広がりやすいか`
- `監査証跡を業務単位で残したいか`

要点:

- 同期 API は実装が単純だが、外部依存の遅延をそのまま受ける
- 非同期化は複雑さが増える代わりに、平準化、再試行、疎結合で強い
- 試験では `即時性` と `耐障害性` のどちらが主役かを見分けると答えやすい

### 追加で見る論点

- `identity federation`
  企業 ID と FM 利用権限をつなぐ
- `Outposts`
  オンプレミス近接でデータを扱いたいとき
- `Wavelength`
  エッジ近傍で低遅延要件を満たしたいとき
- `CodePipeline` / `CodeBuild`
  GenAI 部品の CI/CD、セキュリティスキャン、rollback 支援

### Amazon Q Business を第一候補にする場面

- `社員向けのマネージド生成 AI アシスタント` を早く出したいとき
- `権限制御付きの企業内データ参照`、`引用`、`事前構築コネクタ`、`web experience` が要件に入るとき
- `IAM Identity Center` や `IAM federation` と組み合わせて認証、認可をつなぎたいとき
- プラグインでタスク実行まで広げたいとき

見分け方:

- `Amazon Q Business`
  完成度の高い業務アシスタントを早く提供したい
- `Knowledge Bases`
  自作アプリへ RAG を組み込みたい
- `Agents`
  ツール呼び出しや業務実行まで主役にしたい

## 6. Step Functions が効く場面

- ReAct 的な多段推論を段階管理したい
- 条件分岐やエラー処理を明示したい
- Lambda、Bedrock、各種 API 呼び出しをつなぎたい
- 承認ステップや補償処理を入れたい

要点:

- `自由度は落ちるが、可視性と制御性が高い`
- 安全性、監査性、再試行設計が必要な業務処理では強い

### なぜ Step Functions が頻出か

- 生成AIアプリの失敗は `モデルの誤り` だけでなく `外部 API 失敗`、`承認待ち`、`タイムアウト` でも起きる
- Step Functions は、この `周辺失敗` を吸収する制御面で強い
- つまり、モデル自体の賢さではなく `業務処理として壊れにくくする` 観点で選ばれる

## 7. 配備戦略

### Bedrock 側で見るもの

- オンデマンドでよいか
- Provisioned Throughput が必要か
- Cross-Region inference を許容できるか

### SageMaker AI 側で見るもの

- `Real-time`
- `Serverless`
- `Asynchronous`
- `Batch Transform`

試験では:

- 会話アプリの即時応答 -> `Real-time` または Bedrock のオンライン推論
- 断続利用 -> `Serverless`
- 重い処理 -> `Asynchronous`
- 夜間一括 -> `Batch Transform`

## 8. アプリ統合でよく問われる非機能要件

- タイムアウト
- 再試行
- 冪等性
- レート制御
- 障害隔離
- 監査ログ
- 設定の切り替え

### AppConfig

- プロンプト、モデル切替、しきい値、feature flag の安全な切替に向く
- 本番影響を抑えて設定変更したいときに有効

### Prompt Flows と OpenAPI

- `Prompt Flows`
  ノーコード寄りに prompt chaining を試したいときに向く
- `OpenAPI`
  API 契約を先に固めて、ツール連携や外部公開を標準化したいときに向く

## 9. 引っかけポイント

- 固定手順なのに `Agents` を選ばない
- 単純な API 呼び出しなのに `Step Functions` を過剰導入しない
- `Amazon Q Business` が自然な要件で、わざわざ Bedrock 構成を一から自作しない
- 更新系なのに冪等性と再試行の設計を忘れない
- 高リスク操作なのに human review を省かない
- ストリーミングが必要なのにバッチ前提の設計を選ばない
- `model routing` と `graceful degradation` を混同しない
- エンタープライズ統合なのに `identity federation` や `gateway` を見落とさない

## 10. 判断フロー

1. 推論だけか、実行まで必要か
2. 固定手順か、状況依存か
3. 同期か、非同期か
4. `Amazon Q Business` で足りるか、Bedrock で自作すべきか
5. タイムアウト、再試行、監査をどう入れるか

## Active Recall

- tool use、Agents、Flows の違いは何か
- `A2I` や human augmentation が必要な場面は何か
- Step Functions が有効なのはどんな場面か
- Responses、Converse、InvokeModel はどう使い分けるか
- 企業システム統合で API Gateway、EventBridge、SQS はどう切り分けるか
- `identity federation`、`Outposts`、`Wavelength` はどんな要件で出るか
- `model routing` と `graceful degradation` の違いは何か
- 高リスクな Agent に必要な安全策は何か
- Strands Agents、MCP、Nova Act、AgentCore、A2I は何の論点か

### 答えと解説

1. `tool use` は外部ツール呼び出し、`Agents` は計画とツール選択を含む対話型実行、`Flows` は固定手順のワークフローです。
2. 返金、削除、支払、権限変更など、高リスク操作で人の確認を入れたい場面です。`A2I` や `Step Functions` による human augmentation を疑います。
3. 分岐、再試行、承認、タイムアウト管理が必要な多段処理です。自由度より制御性を優先するときに向きます。
4. 新規会話アプリは `Responses`、複数モデル統一は `Converse`、埋め込みや画像生成は `InvokeModel` です。
5. 即時同期は `API Gateway`、疎結合イベント連携は `EventBridge`、バッファリングや非同期平準化は `SQS` です。
6. `identity federation` は企業 ID と権限統合、`Outposts` はオンプレ近接、`Wavelength` はエッジ近接です。データ所在、低遅延、企業統制の要件で切ります。
7. `model routing` は最適なモデル選択、`graceful degradation` は障害時の縮退運転です。平常時の最適化か、障害時の継続運用かで違います。
8. IAM 最小権限、入力検証、タイムアウト、監査、危険操作の human review です。
9. エージェント実装の周辺技術です。複数エージェント、ツール標準接続、UI 自動化、人手補完、実行基盤を示唆します。
