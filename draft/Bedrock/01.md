## このノートの使い方

このノートは、Amazon Bedrock を次の 4 つの視点で覚えるために作っています。

1. **何ができるか**  
   サービスの機能一覧として理解する。
2. **どんな場面で使うか**  
   要件から機能を逆引きできるようにする。
3. **どうつなぐか**  
   実装時に必要な AWS 部品や API のつながりをイメージできるようにする。
4. **何と何を混同しやすいか**  
   試験で迷いやすい論点を区別できるようにする。

試験では、Amazon Bedrock の機能名を問うより、**ある要件に対してどの構成を選ぶべきか** を問う設問が出やすいです。AIP-C01 の公式試験ガイドも、Foundation Model Integration、Implementation and Integration、AI Safety/Security/Governance、Operational Efficiency and Optimization、Testing/Validation/Troubleshooting という構成で、実装判断を重視しています。[^2]

## 1. まず最初に押さえること

Amazon Bedrock は、単に基盤モデルを呼び出す場所ではありません。AWS 公式の説明では、Bedrock は、主要な AI 企業や Amazon の基盤モデルを統合 API で利用できるフルマネージドサービスであり、モデル選定、推論、RAG、エージェント、ガードレール、評価、カスタマイズ、運用最適化まで含む生成 AI 基盤です。[^1]

### 1.1 Bedrock を 7 層で見る

| 層           | 何を決めるか             | 具体的に考えること                                                                   |
| ------------ | ------------------------ | ------------------------------------------------------------------------------------ |
| モデル層     | どのモデルを使うか       | テキスト、画像、埋め込み、マルチモーダル、速度、コスト                               |
| 推論層       | どの API で呼ぶか        | Responses / Chat Completions / Converse / Invoke                                     |
| 知識拡張層   | 独自データをどう使うか   | Knowledge Bases、chunking、reranking、structured data                                |
| 実行制御層   | どのように処理を進めるか | Agents、Flows、tool use、structured outputs                                          |
| 安全層       | 何を防ぐか               | Guardrails、IAM、PrivateLink、暗号化                                                 |
| 評価層       | 何をどう測るか           | Evaluations、LLM-as-a-judge、人手評価                                                |
| 運用最適化層 | どう安定運用するか       | CountTokens、Prompt caching、Batch、Provisioned Throughput、Inference Profiles、監視 |

この 7 層で見ると、設計時に頭が整理しやすくなります。たとえば、社内 FAQ チャットを作る場合、

- モデル層: テキスト生成モデル
- 推論層: Converse か Responses
- 知識拡張層: Knowledge Bases
- 実行制御層: 単純回答なら Agents までは不要
- 安全層: PII と禁止トピックの Guardrails
- 評価層: groundedness と回答正確性の評価
- 運用最適化層: prompt caching と invocation logging

というように分解できます。

### 1.2 最初に持っておく理解

Amazon Bedrock を学ぶときに、最初から次の整理を持っておくと、かなり迷いにくくなります。

- **モデルを呼ぶだけ**なら、まず推論 API の理解が必要
- **独自データで答えさせる**なら、まず Knowledge Bases を考える
- **外部 API を呼んで業務処理までさせる**なら、Agents を考える
- **処理順が最初から決まっている**なら、Flows を考える
- **不適切出力を防ぐ**なら、Guardrails を考える
- **品質を比べる**なら、Evaluations を考える
- **安定性能や大量処理が必要**なら、Provisioned Throughput や Batch inference を考える

### 1.3 AIP-C01 の試験領域と Bedrock 学習の対応

AWS 公式試験ガイドでは、AIP-C01 の重みづけは次のとおりです。[^2]

| 試験領域                                                                 | 重み | Bedrock で特に対応する論点                                                               |
| ------------------------------------------------------------------------ | ---: | ---------------------------------------------------------------------------------------- |
| Domain 1: Foundation Model Integration, Data Management, and Compliance  |  31% | モデル選定、Knowledge Bases、データ設計、RAG、リージョン、コンプライアンス               |
| Domain 2: Implementation and Integration                                 |  26% | API 使い分け、Agents、Flows、tool use、structured outputs                                |
| Domain 3: AI Safety, Security, and Governance                            |  20% | Guardrails、IAM、PrivateLink、logging、データ保護                                        |
| Domain 4: Operational Efficiency and Optimization for GenAI Applications |  12% | CountTokens、prompt caching、Batch inference、Provisioned Throughput、Inference Profiles |
| Domain 5: Testing, Validation, and Troubleshooting                       |  11% | Evaluations、CloudWatch、CloudTrail、EventBridge、model lifecycle                        |

そのため、学習優先順位としては、まず Domain 1 と 2 に直結する **Knowledge Bases、API の使い分け、Agents / Flows** を固め、その次に Guardrails と最適化、最後に評価・監視・トラブルシュートを詰める流れが覚えやすいです。[^2]

## 2. Bedrock でできることを、具体的に理解する

## 2.1 複数の基盤モデルを統一的に使える

AWS 公式では、Amazon Bedrock は多様な基盤モデルをサポートしており、モデル一覧ページにはプロバイダー、モデル名、モデル ID、対応リージョン、対応モダリティなどが整理されています。[^3][^4]

### 何がうれしいか

単一ベンダーに固定されず、要件ごとにモデルを選び替えられます。

### 具体的なイメージ

- **社内文書 Q&A**: 高精度なテキスト生成モデルを使う
- **類似文書検索**: 埋め込みモデルを使う
- **画像つき商品説明の理解**: 画像とテキストを扱えるモデルを使う
- **画像生成**: 画像生成モデルを使う

### 設計時に見るポイント

- そのモデルは **必要な入力モダリティ** に対応しているか
- そのモデルは **必要な API** に対応しているか
- そのモデルは **必要なリージョン** で使えるか
- そのモデルは **Guardrails、structured output、prompt caching など必要機能** と組み合わせられるか

### 試験での見分け方

問題文に、

- 画像も読みたい
- 埋め込みが必要
- クロスリージョンで使いたい
- 特定リージョン必須

のような条件があれば、モデル名の前に **モダリティ、リージョン、機能互換性** を確認するのが筋です。[^3][^4]

## 2.2 推論 API をどう使い分けるか

AWS 公式の API 概要では、Amazon Bedrock の代表的な推論 API として、Responses API、Chat Completions API、Converse API、Invoke API の使い分けが整理されています。[^5]

### API ごとの理解

| API                  | 公式の位置づけ             | 向いている場面                                  | 具体例                                         |
| -------------------- | -------------------------- | ----------------------------------------------- | ---------------------------------------------- |
| Responses API        | 新しく始める用途の推奨 API | OpenAI 互換、ステートフル会話、サーバー側ツール | 新規のチャットアプリ、エージェント的な UI      |
| Chat Completions API | 軽量なステートレス会話     | OpenAI 互換、会話履歴を自前管理                 | 既存 OpenAI 系実装の移植                       |
| Converse API         | 統一インターフェース       | Bedrock 標準で複数モデルをまたぎたい            | Claude と Nova を同じコードで差し替える        |
| Invoke API           | 直接制御                   | モデル固有パラメータ、画像生成、埋め込みなど    | 埋め込み生成、画像生成、モデル固有フォーマット |

### どう選ぶか

#### Responses API

**こういうときに使う**

- OpenAI 互換で始めたい
- 会話状態をできるだけ Bedrock 側に寄せたい
- サーバー側ツールを使いたい

**具体例**

- カスタマーサポート用チャット UI
- 社内アシスタントで会話文脈を継続したい

#### Chat Completions API

**こういうときに使う**

- 既存アプリが OpenAI の chat completions を前提にしている
- 会話履歴を DynamoDB やアプリ側 DB で明示的に管理したい
- テキスト中心で軽量にやりたい

**具体例**

- 既存 SaaS の問い合わせ欄にチャット補助を埋め込む
- 会話の保存・削除を自社ルールで厳密に制御したい

#### Converse API

**こういうときに使う**

- Bedrock 標準の書き方で統一したい
- 複数モデルを後で差し替える可能性が高い
- メッセージベースで素直に書きたい

**具体例**

- PoC で Claude と Nova を比較しながら検証する
- モデル切り替え機能をアプリ側に持たせたい

#### Invoke API

**こういうときに使う**

- 埋め込み、画像生成、単発推論を扱う
- モデル固有の request/response をそのまま使いたい
- Converse の統一フォーマットに寄せるより、生の制御を優先したい

**具体例**

- 埋め込みを作ってベクトル DB へ入れる
- 画像生成モデルへプロンプトを送る

### 実装の見え方

#### 会話アプリの最小構成

- フロントエンド
- API Gateway または App Runner / Lambda
- Bedrock Responses or Converse
- CloudWatch / logging

#### 埋め込み生成の最小構成

- バッチ処理 or Lambda
- Bedrock InvokeModel
- ベクトルストア

### 迷ったときの判断

- **新規のチャットアプリ** → まず Responses API を検討[^5]
- **複数モデルを同じコードで扱いたい** → Converse API[^5]
- **画像生成や埋め込み** → Invoke API[^5]
- **OpenAI 系資産を移植したい** → Chat Completions API[^5]

## 2.3 Prompt management は、プロンプトを運用資産にする機能

Prompt management では、プロンプトを保存し、変数を持たせ、推論パラメータを含め、再利用できる形で管理できます。[^6]

### 何ができるか

- プロンプトを保存する
- 変数を埋め込む
- モデルやパラメータを紐づける
- バージョンを切る
- 複数バリアントを比較する

### 具体例

#### 例 1: 問い合わせ返信プロンプト

同じ用途でも、

- 丁寧さ重視版
- 簡潔さ重視版
- 技術説明重視版

のように複数バリアントを持てます。

#### 例 2: 要約プロンプト

変数として、

- `audience`: 経営層 / 開発者 / CS
- `max_length`: 100 文字 / 300 文字
- `language`: ja / en

を入れて、1 つのプロンプト定義を使い回せます。

### 実務での使い方

- コンソール上でたたき台を作る
- バリアント比較で出力差を見る
- 本番採用版を version として固定する
- アプリは draft ではなく version を呼ぶ

### ありがちな失敗

- プロンプトをソースに直書きしすぎて差分管理しづらくなる
- 調整中の draft を本番で参照してしまう
- 出力差の比較をしないまま勘で採用する

### 試験での見分け方

問題文に、

- 複数プロンプトの再利用
- バージョン管理
- A/B 比較
- 業務フロー間で共通利用

が出てきたら、Prompt management が候補です。[^6]

## 2.4 Tool use と structured outputs で、生成結果を業務処理につなぐ

Bedrock では tool use と structured outputs が使えます。structured outputs では JSON Schema を使って検証可能な JSON を返させることができ、tool use ではモデルがツール呼び出し要求を返し、その結果を再投入して最終回答を作れます。[^7][^8]

### 何が変わるか

単なる自然文生成から、**機械が後続処理しやすい生成** に変わります。

### 具体例 1: 問い合わせ分類

ユーザー文:

- 配送が遅い
- 返金してほしい
- ログインできない

これを自然文で返させるのではなく、次のような JSON にさせます。

```json
{
  "category": "refund",
  "priority": "high",
  "requires_human": true
}
```

この形式なら、下流で

- `category=refund` なら返金フローへ送る
- `priority=high` なら即時通知する
- `requires_human=true` なら有人対応へ送る

という処理につなげられます。

### 具体例 2: 受注確認 API 呼び出し

モデルが会話の中で、

- orderId が必要
- customerEmail が必要

と判断したら、それをユーザーに聞き出し、必要情報がそろった時点で `GetOrderStatus` ツール呼び出しを返します。

### OpenAPI スキーマの最小イメージ

```yaml
openapi: 3.0.0
info:
  title: Order API
  version: 1.0.0
paths:
  /orders/{orderId}:
    get:
      operationId: GetOrderStatus
      parameters:
        - name: orderId
          in: path
          required: true
          schema:
            type: string
        - name: customerEmail
          in: query
          required: true
          schema:
            type: string
      responses:
        "200":
          description: order status
```

### 実務でのポイント

- 人が読むだけなら自然文でもよい
- システム連携があるなら JSON Schema で縛る
- 外部処理が必要なら tool use または Agents を考える

### 試験での見分け方

- **JSON で返して後続システムへ渡したい** → structured outputs[^7]
- **外部 API を呼びたい** → tool use または Agents[^8]
- **会話の中で不足情報を埋めたい** → Agents がより自然[^9]

## 2.5 Knowledge Bases は、独自データを検索して答えるための中核機能

Knowledge Bases は Bedrock の RAG 機能です。公式ドキュメントでは、検索、自然言語応答生成、引用、structured data 向けの自然言語から SQL への変換、データ同期、reranking、Agents への組み込みなどが説明されています。[^10]

### まず全体像

Knowledge Bases の基本の流れは、次のように理解すると実装が見えます。

1. データソースをつなぐ[^10]
2. 取り込み時に parsing と chunking を行う[^11]
3. 埋め込みに変換してベクトルインデックスへ格納する[^11]
4. 問い合わせ時に関連チャンクを検索する[^10]
5. 必要なら reranker で並び替える[^10]
6. 取得結果を使って応答生成する[^10]
7. 必要なら引用を返す[^10]

### 取り込みの実装イメージ

#### 非構造データ

- S3 の PDF、Markdown、HTML
- Web crawler
- Confluence
- SharePoint
- Salesforce
- Custom data source

#### 構造化データ

- 構造化データストアへ自然言語から問い合わせし、SQL などへ変換して取得[^10]

### chunking はなぜ重要か

AWS 公式では、Knowledge Bases は取り込み時に文書を chunk に分割し、embedding に変換して vector index に書き込みます。[^11]

これが重要なのは、検索の最小単位が chunk になるからです。

#### 具体的に起こること

- 1 つの chunk が大きすぎる  
  → 検索で余計な文脈まで混ざる
- 1 つの chunk が小さすぎる  
  → 必要な前後関係が切れて回答が弱くなる

#### イメージしやすい例

設計書に、

- 概要
- 前提条件
- エラーハンドリング
- サンプルレスポンス

が連続して書かれているとします。

このとき、エラー条件だけを知りたい質問に対して、chunk が大きすぎると回答生成時に余分な概要文まで大量に入ってきます。逆に小さすぎると、エラーコードだけ見つかって、その意味や対処が別 chunk に分かれてしまいます。

### vector store をどう考えるか

AWS 公式の Knowledge Bases の説明では、利用する vector store のセットアップや Bedrock コンソールによる OpenSearch Serverless の自動作成、また外部 vector store の利用が説明されています。サポート対象として、OpenSearch Serverless、Neptune、Aurora (RDS)、Pinecone、Redis Enterprise Cloud などが案内されています。[^10][^12][^13]

#### 試験で大事な理解

- Knowledge Bases は **埋め込みモデル + vector store + retrieval + generation** の組み合わせ
- Bedrock がすべてを魔法のようにやるわけではなく、vector index という考え方がある
- embeddings の次元や store 互換性が設計に影響する[^12]

### reranker を入れる場面

最初の検索で 10 件ほど取れても、その並び順が業務的な関連度に十分でないことがあります。

#### 具体例

問い合わせが「管理者画面の MFA 設定手順を教えて」なのに、

- MFA の概念説明
- 一般ユーザー向け MFA
- 管理者向け画面手順

が混在して返ると、生成時に適切な文脈が弱まります。

このとき reranker を挟むと、より質問に近いチャンクを上位へ寄せやすくなります。[^10]

### multimodal Knowledge Bases のイメージ

AWS 公式では、Knowledge Bases はマルチモーダルコンテンツにも対応しており、画像、音声、動画を扱えます。画像をクエリにでき、音声や動画では timestamp 付きで relevant segment を返せます。[^14]

#### 具体例

- 商品カタログ画像を見せて類似商品を検索する
- 研修動画の中から「MFA 設定」の説明箇所だけ見つける
- 通話録音から「解約したい」が出た部分だけ取り出す

### structured data に対する使い方

Knowledge Bases は、structured data store に対して自然言語から SQL のような問い合わせへ変換して取得できます。[^10]

#### 具体例

ユーザーが、

- 先月の返品件数は？
- 地域別売上の上位 3 件は？

と聞くと、Knowledge Base 側が構造化データストア向けの問い合わせへ変換して必要情報を取得します。

### 実務でよくある構成

#### 社内ドキュメント検索チャット

- S3 / Confluence
- Knowledge Base
- 埋め込みモデル
- vector store
- reranker
- text generation model
- citations
- Guardrails

#### 期待する振る舞い

- 回答の根拠を返す
- 未記載なら「わからない」と言う
- 余計な推測を減らす

### Knowledge Bases を選ばないほうがよい場面

- データ検索ではなく API 実行が主目的
- タスク進行の会話制御が主目的
- 固定順のワークフローが主目的

この場合は Agents や Flows のほうが中心になります。

### 試験での見分け方

- **独自データに基づいて答える** → Knowledge Bases[^10]
- **引用が必要** → Knowledge Bases[^10]
- **更新データを反映したい** → Fine-tuning より Knowledge Bases を優先して考える[^10]
- **structured data を自然言語で引きたい** → Knowledge Bases の structured data 連携を考える[^10]

## 2.6 Agents は、会話しながらタスク完了まで進める

AWS 公式では、Agents は、指示、アクショングループ、Knowledge Bases、必要に応じて Lambda や API を組み合わせ、タスク実行まで進める仕組みとして説明されています。アクショングループでは OpenAPI schema または function detail schema を使えます。[^9][^15]

### Agents を 1 文で言うと

**ユーザーの意図を読み、足りない情報を聞き、必要なら API を呼んで、最後まで仕事を進める仕組み** です。

### 構成要素

- 指示文
- モデル
- action group
- OpenAPI schema または function detail schema[^9]
- optional な Lambda executor[^9]
- optional な Knowledge Base[^10]

### 具体例: 予約変更エージェント

ユーザー: 明日の予約を 15 時に変更したい

エージェントがやること:

1. 予約番号が必要か判断する
2. 足りなければ聞く
3. 顧客確認情報が必要なら聞く
4. `ChangeReservation` API を呼ぶ
5. 結果を会話文として返す

#### このとき必要になりやすい部品

- 予約 API
- OpenAPI schema
- Lambda executor
- 予約ポリシーを格納した Knowledge Base
- Guardrails

### function detail schema を使うイメージ

function detail schema は、API エンドポイントを厳密に表現するより、まず必要なパラメータ群を agent に理解させたいときに向いています。[^9]

### Lambda を入れる場面

AWS 公式では、Lambda は、エージェントが決定した API operation や parameters を受け取り、その結果を返す実行部として使えます。[^9][^16]

#### 具体例

- 複数の内部 API をまとめて呼ぶ
- 独自認証トークンを付ける
- 業務ルールを事前チェックする
- 既存 SOAP / legacy API へ変換して投げる

### code interpretation を使う場面

code interpretation は、コード生成・実行・トラブルシュートが必要な処理に使います。[^17]

#### 具体例

- CSV から集計表を作る
- 計算処理を挟んで回答する
- 生成した表を元に分析する

### Agents を選ぶ条件

- 会話中に **不足情報を補う** 必要がある
- API 呼び出しなど **行動** が必要
- 1 問 1 答ではなく **タスク完了** が目的

### Agents を選ばないほうがよい条件

- ただの検索回答なら Knowledge Bases のほうが自然
- 処理順が決まっているなら Flows のほうが見通しがよい
- ただの JSON 出力なら structured outputs だけで十分なことがある

### 試験での見分け方

- **予約変更、注文確認、申請処理** のような業務語が出る
- **必要情報を聞き出す** とある
- **API を呼ぶ** とある

この 3 つがそろうと、かなり Agents 寄りです。[^9][^15]

## 2.7 Flows は、決まった順序の生成 AI ワークフローを組む

Flows は、モデル、プロンプト、Knowledge Bases、guardrail などをノードとしてつなぎ、視覚的または API でワークフローを構築する機能です。公式にはテンプレートや example flows もあります。[^18][^19]

### Flows の本質

**順序と分岐が先に決まっている処理を、見える形で組む** 機能です。

### 具体例: 問い合わせ triage フロー

1. 入力文を受け取る
2. 分類ノードでカテゴリ判定
3. 技術問い合わせなら Knowledge Base ノードへ送る
4. 請求問い合わせなら別のプロンプトノードへ送る
5. 最後に JSON 出力ノードへまとめる

### こういう場面で使いやすい

- 処理順が業務フローとして明確
- 誰が見ても流れが分かるようにしたい
- エージェントほど自由な意思決定は不要

### Agents との違いを具体で見る

#### Agents 向き

- ユーザーとの会話に応じて、次に聞くことが変わる
- 実行する action が状況で変わる

#### Flows 向き

- 入力を分類して、決まったノードへ送る
- 一定の順序で処理する
- 視覚的な保守性を重視する

### Guardrails との連携

AWS 公式では、Flows 内の node に Guardrails を含める設定も案内されています。[^20]

### 試験での見分け方

- **分類 → 検索 → 要約 → 出力** のように順序が見えている
- **可視的にワークフロー管理したい**
- **ノード接続** の文脈

こういうときは Flows がかなり本命です。[^18]

---

## 2.8 Guardrails は、入力と出力の両方を安全制御する

AWS 公式では、Guardrails は user input と model response の両方を評価して、安全性を高める機能です。1 アカウント内に複数の Guardrail を持てて、用途ごとに設定できます。[^21]

### Guardrails で考えるべきこと

- 何を止めたいか
- 何を伏せたいか
- 何に基づかない回答を防ぎたいか

### 具体的な設定イメージ

#### 1. PII 保護

- 氏名
- メールアドレス
- 電話番号
- 個人識別情報

#### 2. 禁止トピック

- 医療診断
- 投資助言
- 社外秘業務内容

#### 3. 単語フィルタ

- 不適切なワード
- NG 用語
- ブランド上避けたい表現

#### 4. contextual grounding

- 検索した文書の範囲から外れる推測を減らしたい
- 根拠のない断定を抑えたい

### 具体例: 社内規程ボット

- 規程に書いていないことは断定しない
- 個人情報をそのまま出さない
- 人事・法務の禁止質問は回答を制限する

### ApplyGuardrail API の使いどころ

推論そのものとは別に、入力や出力に Guardrail を単独適用したい場面があります。[^22]

#### 具体例

- ユーザー投稿を保存前に検査する
- 別の LLM の出力を最後に検査する
- 生成以外のテキストパイプラインにも同じポリシーをかける

### 試験での見分け方

- **不適切出力を防ぎたい** → Guardrails
- **PII を検出・保護したい** → Guardrails
- **根拠外回答を減らしたい** → contextual grounding を伴う Guardrails

Guardrails は安全制御であって、品質比較そのものではありません。そこは Evaluations と切り分けます。[^21][^23]

## 2.9 Evaluations は、モデルや RAG の品質を比べる

AWS 公式では、Bedrock Evaluations は、Bedrock 内のモデルや knowledge bases だけでなく、Bedrock 外のモデルや RAG source も評価できると説明されています。[^23]

### Evaluations で何を見るか

- どのモデルが回答品質でよいか
- どのプロンプト版がよいか
- Knowledge Base の retrieval と response が妥当か
- 安全性や頑健性が十分か

### 具体例 1: モデル比較

100 件の問い合わせセットを用意して、

- Claude 系モデル
- Nova 系モデル

で比較し、

- 正確性
- 有用性
- 冗長さ
- groundedness

を見ます。

### 具体例 2: Knowledge Base 比較

同じデータに対して、

- chunking 設定 A
- chunking 設定 B
- reranker なし
- reranker あり

を比べて、どれが引用の質と回答精度が高いかを検証します。

### LLM-as-a-judge と human-based evaluation

AWS 公式では、LLM judge と human-based evaluation の両方が使えます。[^24][^25]

#### LLM-as-a-judge 向き

- 大量比較
- 初期スクリーニング
- 継続評価

#### human-based evaluation 向き

- 最終判断
- ドメイン専門性が必要
- 微妙な語調やブランド適合性を見る

### 試験での見分け方

- **どのモデルがよいか比較したい** → Evaluations
- **RAG の正しさを測りたい** → Evaluations
- **安全制御ではなく品質測定が目的** → Evaluations

## 2.10 モデルカスタマイズは、何を変えたいかで手法が変わる

Bedrock では、Fine-tuning、continued pre-training、distillation、reinforcement fine-tuning、custom model import が用意されています。[^26][^27][^28][^29][^30]

### まず結論

- **知識を外から参照したい** → RAG
- **モデルの出力傾向を変えたい** → fine-tuning
- **ドメイン知識をより深く染み込ませたい** → continued pre-training
- **より小さく速くしたい** → distillation
- **報酬設計で改善したい** → reinforcement fine-tuning
- **外で育てたモデルを持ち込みたい** → custom model import

### Fine-tuning

#### 何を変えるか

- 返答スタイル
- 分類精度
- 出力フォーマットの一貫性
- 業務固有ラベルへの追従

#### 具体例

- 問い合わせ分類のラベルを社内定義に合わせたい
- 医療文書ではなく、社内保守チケットの文体に寄せたい

### Continued pre-training

#### 何を変えるか

- ドメイン語彙や文脈へのなじみ方
- 長期的な専門知識の内部化

#### 具体例

- 法務、製薬、半導体のような専門文書群に大量適応したい

### Distillation

#### 何を変えるか

- 高性能な教師モデルのふるまいを小さい学生モデルへ移す

#### 具体例

- 高精度モデルで作った出力を教師にして、安価なモデルへ学習させる
- 高トラフィック向けにコストを下げたい

### Reinforcement fine-tuning

#### 何を変えるか

- 報酬関数に沿った応答方向

#### 具体例

- できるだけ短く正確に答える応答を強く好む
- 安全性ポリシーを強く守る応答を高報酬にする

### Custom model import

#### 何を変えるか

- Bedrock 外で用意したカスタムモデルを Bedrock 運用へ持ち込む

#### 具体例

- 既に別の ML 基盤で調整済みのモデルを Bedrock の周辺機能と合わせて使いたい

### 実務での注意

AWS 公式では、カスタマイズ済みモデルを使うには Provisioned Throughput が必要です。[^31]

### 試験での見分け方

- **最新文書を見て答えたい** → Fine-tuning ではなく RAG を優先
- **文体やラベルを安定させたい** → Fine-tuning
- **大規模な専門コーパスをなじませたい** → Continued pre-training
- **性能を残しつつ軽量化したい** → Distillation
- **外で育てたモデル資産がある** → Import

## 2.11 コスト、性能、スループットを最適化する

Bedrock には CountTokens、prompt caching、batch inference、Provisioned Throughput、inference profiles、cross-Region inference など、運用最適化のための機能があります。[^32][^33][^34][^35][^36][^37]

### CountTokens

AWS 公式では、CountTokens API は、同じ入力を実際に推論した場合の token count を事前に返します。[^32]

#### 何に効くか

- コスト見積もり
- quota 見積もり
- 長すぎる入力の事前検知

#### 具体例

- 10 ページの議事録を要約したい前に token 数を測る
- FAQ 全文を毎回 prompt に入れる前に大きさを確認する

### Prompt caching

AWS 公式では、prompt caching は、supported model に対してレイテンシーと input token cost を下げる機能です。[^33]

#### こういうときに効く

- system prompt が毎回ほぼ同じ
- 長い業務マニュアルを共通で渡している
- 会話履歴の前半が固定に近い

#### 具体例

- 20KB の業務ルールを毎リクエストで送っている社内ボット
- 同じ長いプロダクト仕様書を参照しながら複数質問を受けるアプリ

#### 逆に効きにくい場面

- 毎回まったく違う長文入力
- 共通コンテキストが短い

### Batch inference

AWS 公式では、batch inference は、複数 prompt を S3 input/output で非同期に処理する仕組みです。[^34]

#### 具体例

- 5 万件の商品レビュー要約
- 1 万件のサポートチケット分類
- 録音ファイルの一括要約

#### リアルタイム API と違う点

- すぐ返らない
- 夜間ジョブに向く
- S3 ベースで回す

### Provisioned Throughput

AWS 公式では、Provisioned Throughput は固定コストで高い throughput を確保する仕組みで、カスタムモデル利用時は必須です。削除するまで課金が継続します。[^31][^35]

#### 向く場面

- 本番トラフィックが安定して高い
- p95 レイテンシーを安定させたい
- custom model を使う

#### 具体例

- 社外公開の問い合わせ窓口
- 社内全社向けアシスタント
- キャンペーン期間中に大量アクセスが確実に来る

### Inference profiles

AWS 公式では、inference profile は、モデルと 1 つ以上のリージョンを束ね、使用量追跡、コスト追跡、cross-Region inference に使う resource です。[^36]

#### application inference profile が便利な場面

- アプリ単位で cost allocation tag を付けたい
- チーム別利用量を追いたい
- 同じモデルを複数アプリで分けて管理したい

### Cross-Region inference

AWS 公式では、cross-Region inference には Geographic と Global の 2 種類があり、地理境界内で処理したいか、最大 throughput と cost/performance を優先するかで選びます。[^37]

#### 具体的な考え方

- **Geographic**: US / EU / APAC など地理境界を守りたい
- **Global**: 地域制約が薄く、最大 throughput とコスト最適化を優先したい

#### 注意点

- cross-Region inference は inference profile 経由で使う[^36][^37]
- Provisioned Throughput は cross-Region inference では使わない[^37]

### 実務での選び方を一気に整理

| 要件                                   | 第一候補                      | 理由                         |
| -------------------------------------- | ----------------------------- | ---------------------------- |
| 入力が長すぎないかを事前確認したい     | CountTokens                   | 事前見積もり                 |
| 同じ長い文脈を何度も使う               | Prompt caching                | 入力再計算を減らす           |
| 大量の非同期処理                       | Batch inference               | S3 ベースで一括処理          |
| 高トラフィックで安定性能が必要         | Provisioned Throughput        | 専有 throughput              |
| アプリ単位で usage / cost を追いたい   | Application inference profile | 追跡しやすい                 |
| 複数リージョンで throughput を上げたい | Cross-Region inference        | 複数リージョンへルーティング |

### 試験での見分け方

- **大規模一括処理** → Batch inference[^34]
- **長い共通 prompt** → Prompt caching[^33]
- **専有性能が必要** → Provisioned Throughput[^35]
- **アプリ別コスト追跡** → Application inference profile[^36]
- **地理制約を守りつつ throughput を上げたい** → Geographic cross-Region inference[^37]

## 2.12 セキュリティ、監査、監視

AWS 公式では、Bedrock は、暗号化、PrivateLink、IAM、CloudWatch、CloudTrail、EventBridge などと組み合わせて監視・監査できます。[^38][^39][^40][^41]

### データ保護

AWS 公式の Bedrock 概要では、セキュリティ、プライバシー、責任ある AI の機能が Bedrock の中核として示されています。[^1]

### VPC / PrivateLink

PrivateLink を使うことで、Bedrock への通信をプライベート接続にできます。[^39]

#### 具体例

- 金融・医療系でパブリック経路を避けたい
- 社内閉域網前提の設計にしたい

### Model invocation logging

AWS 公式では、model invocation logging を有効にすると、CloudWatch Logs や S3 にリクエスト／レスポンスを出力できます。[^40]

#### 実務で役立つ場面

- なぜ不適切な回答が出たか調べる
- コストの多い prompt を探す
- モデル変更前後で出力差を確認する

### CloudWatch メトリクス

Bedrock の監視ページには runtime metrics が整理されており、InvocationLatency、InvocationClientErrors、InvocationServerErrors、InvocationThrottles、InputTokenCount、OutputTokenCount などを見られます。[^41]

#### 具体的な見方

- `InvocationLatency` が上がる  
  → 入力肥大化、モデル変更、混雑、周辺処理増加を疑う
- `InvocationThrottles` が出る  
  → quota、burst、cross-Region inference の検討余地
- `InputTokenCount` が急増する  
  → prompt や会話履歴が膨らんでいる
- `LegacyModelInvocations` が出る  
  → 移行漏れを疑う[^41][^42]

### CloudTrail / EventBridge

AWS 公式では、Bedrock API calls の監査に CloudTrail、job state changes の監視に EventBridge が使えます。[^38]

#### 具体例

- batch inference job 完了で後続処理を起動する
- モデル変更操作を監査する
- guardrail や knowledge base の操作履歴を確認する

### モデルライフサイクル

AWS 公式では、モデルは Active / Legacy / EOL の状態を持ち、少なくとも 12 か月は提供され、Legacy は少なくとも 6 か月続きます。EOL 後は自動移行されません。[^42]

#### 実務で重要な点

- モデル名固定でコードを書きっぱなしにしない
- ListFoundationModels / GetFoundationModel やドキュメントで lifecycle を確認する
- 移行計画を持つ

### 試験での見分け方

- **安全接続** → PrivateLink
- **誰が何をしたかの監査** → CloudTrail
- **推論ログ分析** → model invocation logging + CloudWatch / S3
- **遅延やスロットル監視** → CloudWatch metrics
- **古いモデルの継続利用リスク** → model lifecycle

## 2.13 Bedrock Data Automation は、非構造データを構造化する

AWS 公式では、BDA は document、image、video、audio のような非構造データから insight を抽出し、構造化形式へ変換するためのサービスです。Intelligent Document Processing、classification、extraction、normalization、validation の自動化に使えます。[^43][^44][^45][^46]

### 何がうれしいか

会話生成ではなく、**業務データ化** が主目的のときに、かなり筋がよいです。

### 具体例 1: 請求書処理

抽出したい項目:

- 請求書番号
- 発行日
- 支払期日
- 金額
- 仕入先名

BDA の blueprint で、抽出したい field を定義します。AWS 公式でも、blueprint で抽出したい field を指定して document から取り出す流れが説明されています。[^44]

### 具体例 2: 契約書の正規化

文書ごとに、

- 契約相手名
- 契約開始日
- 契約終了日
- 自動更新有無

の表記がバラバラでも、normalization で一定形式にそろえられます。[^46]

### 具体例 3: 通話録音や動画の処理

- 音声をもとに発話内容を抽出する
- 動画の特定場面を構造化する
- 監査ログやサマリを作る

### BDA と Knowledge Bases の違い

- **BDA**: まず非構造データを業務で使いやすい structured output にする
- **Knowledge Bases**: 検索して回答する

### 試験での見分け方

- **会話ではなく抽出・分類・正規化が目的** → BDA
- **文書からフィールドを抜きたい** → BDA
- **動画や音声の内容を structured output にしたい** → BDA

## 2.14 Bedrock Marketplace は、モデル候補を広げる

AWS 公式では、Bedrock Marketplace を通じて追加のモデルを見つけ、サブスクライブし、Bedrock API から利用できます。Agents、Knowledge Bases、Guardrails と統合できると案内されています。[^47]

### 具体例

- 金融、法務、画像解析などの特化モデルを試したい
- 標準モデルでは要件に合わない分野特化モデルを使いたい

### どう覚えるか

Bedrock は AWS 提供モデルだけの箱ではなく、**周辺の統合機能まで含めてモデル選択肢を広げる基盤** だと理解するとよいです。[^47]

## 3. ユースケース別に、構成を組み立てる

## 3.1 社内ドキュメント検索チャット

### 要件

- 社内規程、設計書、FAQ に基づいて回答
- 根拠を示したい
- 最新文書を反映したい
- 個人情報は出したくない

### 第一候補構成

- text generation model
- Knowledge Base
- embedding model
- vector store
- reranker
- citations
- Guardrails
- CloudWatch / logging

### 実装の流れ

1. S3 か Confluence を data source にする[^10]
2. Knowledge Base を作る[^10]
3. chunking を調整する[^11]
4. 必要に応じて reranker を入れる[^10]
5. Retrieve-and-generate で引用つき回答を返す[^10]
6. PII と禁止トピックを Guardrails で制御する[^21]

### なぜ fine-tuning ではないか

最新文書の差し替えと引用が主目的なので、まず RAG のほうが自然です。fine-tuning は知識更新のたびに再学習する筋ではありません。[^10][^26]

### ありがちな失敗

- chunking を適当にして retrieval 精度が落ちる
- citations を返さず検証不能になる
- 社内文書未記載の内容をモデルが推測してしまう

---

## 3.2 予約・受注・申請を処理する業務エージェント

### 要件

- 会話で依頼を受ける
- 不足情報を聞く
- API を呼んで手続きを完了する
- 業務ルールを守る

### 第一候補構成

- Agent
- action groups
- OpenAPI schema
- Lambda
- optional Knowledge Base
- Guardrails

### 実装の流れ

1. Agent instruction を作る[^9]
2. `CreateReservation`, `CancelReservation`, `GetStatus` など action group を用意する[^15]
3. OpenAPI schema を置く[^15]
4. Lambda で既存 API と接続する[^16]
5. ポリシーや約款を参照するなら Knowledge Base を付ける[^10]
6. 個人情報、禁止回答を Guardrails で制御する[^21]

### この構成が自然な理由

- 回答だけで終わらず、処理完了まで進められる
- 必要情報を聞き出せる
- 既存 API とつながる

### 試験でのキーワード

- 予約変更
- 注文確認
- 顧客情報確認
- API 実行
- 必須パラメータを収集

---

## 3.3 固定順の審査フローや文書処理フロー

### 要件

- 手順が最初から決まっている
- ノード単位で責務を分けたい
- 可視的に保守したい

### 第一候補構成

- Flows
- Prompt management
- Knowledge Base
- Guardrails
- structured outputs

### 実装例

1. 入力文を受ける
2. 分類ノードで問い合わせ種別を決める
3. 種別ごとに別 prompt を使う
4. 必要なら Knowledge Base で補強する
5. JSON で出力する
6. 後段システムへ送る

### Agents ではなく Flows が自然な理由

- 会話で次の行動を自律決定する必要がない
- 決まった手順を守ることが大事
- ノード構成の説明責任がある

---

## 3.4 大量のレビュー、チケット、議事録を夜間一括処理する

### 要件

- 数万件単位
- リアルタイム不要
- S3 入出力でよい

### 第一候補構成

- Batch inference
- S3
- Application inference profile
- CloudWatch / EventBridge

### 実装例

1. 入力 JSONL やテキスト群を S3 に置く[^34]
2. Batch inference job を作成する[^34]
3. 完了イベントで後続処理を起動する[^38]
4. 必要なら評価セットで品質確認する[^23]

### 試験でのキーワード

- 夜間処理
- 数万件
- 非同期
- コスト効率
- リアルタイム不要

---

## 3.5 長い共通文脈を何度も使うアシスタント

### 要件

- 長い system prompt が毎回ほぼ同じ
- 業務ルールを共通で渡している
- レイテンシーと入力コストを下げたい

### 第一候補構成

- Prompt caching
- CountTokens
- Converse or Responses

### 実装例

1. CountTokens で入力サイズを把握する[^32]
2. 共通コンテキスト部分を cache 対象にする[^33]
3. 会話差分だけを都度追加する

### 試験でのキーワード

- 毎回同じ長文 prompt
- 入力コスト削減
- レイテンシー改善

---

## 3.6 厳密な構造化データ抽出

### 要件

- PDF、画像、音声、動画から field を抜きたい
- 人が読む回答より downstream 連携が大事

### 第一候補構成

- BDA
- Blueprints
- S3
- 後続 DB / 業務システム

### 実装例

1. 抽出したい field を blueprint で定義する[^44]
2. BDA に処理させる[^43]
3. 必要なら normalization / transformation をかける[^45][^46]
4. JSON として後段へ渡す

### 試験でのキーワード

- extraction
- normalization
- classification
- document processing
- structured output from unstructured data

## 5. 試験で混同しやすい論点

### 5.1 RAG と fine-tuning

- **RAG**: 外部知識をその都度参照する[^10]
- **Fine-tuning**: モデルの出力傾向を変える[^26]

### 5.2 Knowledge Bases と Agents

- **Knowledge Bases**: 検索して答える[^10]
- **Agents**: 必要情報を集めて action を実行する[^9]

### 5.3 Agents と Flows

- **Agents**: 次の行動を動的に決める[^9]
- **Flows**: あらかじめ決めた流れを実行する[^18]

### 5.4 Guardrails と Evaluations

- **Guardrails**: 実行時の安全制御[^21]
- **Evaluations**: 事前・継続的な品質比較[^23]

### 5.5 Prompt caching と fine-tuning

- **Prompt caching**: 同じ長文入力の再利用最適化[^33]
- **Fine-tuning**: モデルのふるまい自体を変える[^26]

### 5.6 Batch inference と Provisioned Throughput

- **Batch inference**: 非同期の大量処理[^34]
- **Provisioned Throughput**: 同期推論の安定性能確保[^35]

### 5.7 Application inference profile と cross-Region inference profile

- **Application inference profile**: usage / cost tracking が主眼[^36]
- **Cross-Region inference profile**: 複数リージョンへルーティングして throughput 向上が主眼[^37]

- prompt を資産として管理する感覚を持つ

目的:

- RAG の一連の流れを理解する
- chunking と retrieval の感覚を持つ

### Step 5. structured outputs を試す

目的:

- 自然文ではなく業務連携用の JSON を返す設計を体験する

### Step 6. Agent で API 実行までつなぐ

目的:

- どこから「ただのチャット」ではなくなるかを体感する

### Step 7. Guardrails と Evaluations を入れる

目的:

- 安全制御と品質評価を切り分けて理解する

### Step 8. Prompt caching / Batch / Inference profile を試す

目的:

- 本番運用で見るべき論点を学ぶ

### Step 9. logging と CloudWatch で監視する

目的:

- 生成 AI は答えるだけではなく、運用して初めて完成することを理解する

## 7. 運用で見ておきたい観測点

### 7.1 レイテンシーが高いとき

まず見るもの:

- InvocationLatency[^41]
- InputTokenCount[^41]
- モデル切り替え有無
- prompt 増大
- reranker や KB の追加有無

### 7.2 スロットルが出るとき

まず見るもの:

- InvocationThrottles[^41]
- リクエスト集中時間帯
- cross-Region inference の余地[^37]
- Provisioned Throughput の必要性[^35]

### 7.3 品質がぶれるとき

まず見るもの:

- prompt version の変化[^6]
- model version の変化[^42]
- KB の chunking 設定[^11]
- retrieval result の質[^10]
- reranker の有無[^10]

### 7.4 コストが増えたとき

まず見るもの:

- InputTokenCount / OutputTokenCount[^41]
- prompt の肥大化
- 会話履歴の持ちすぎ
- prompt caching 未利用[^33]
- batch 化できる処理が同期で回っていないか[^34]

### 7.5 監査や原因調査をしたいとき

まず見るもの:

- model invocation logging[^40]
- CloudTrail[^38]
- job state change events[^38]

## 8. 最後に、Bedrock を一文で言い切る

Amazon Bedrock は、

- 複数モデルの **推論基盤** であり、
- 独自データ活用の **RAG 基盤** であり、
- API 実行や業務処理まで進める **エージェント基盤** であり、
- 安全制御、評価、監視、コスト最適化まで含めた **運用基盤**

です。[^1][^10][^21][^23][^35]

試験では、何があるかを覚えるだけでは足りません。  
**この要件なら、なぜその機能を選ぶのか** を説明できる状態が、合格ラインに近い理解です。[^2]
