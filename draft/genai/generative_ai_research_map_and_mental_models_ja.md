# 生成AIを理解するための研究地図とメンタルモデル
**副題:** LLM / アラインメント / RAG・エージェント / 画像生成 / 評価・安全性 / test-time scaling  
**作成日:** 2026-03-09  
**言語:** 日本語

---

## 0. この文書の目的

本資料は、ここまでの議論をもとに、生成AIを理解するための「研究地図」と「メンタルモデル」を一つの体系にまとめたものです。  
狙いは、個別の論文や流行語を追いかけることではなく、**新しいモデルや研究が出てきたときに、それが技術スタックのどこを前進させたものかを見分けられる頭の型を作ること**にあります。

本資料の基本姿勢は次の三つです。

1. 生成AIを**単体のモデル**としてではなく、**訓練・推論・検索・ツール・評価を含むシステム**として理解する。  
2. 生成AIを**万能な知性**としてではなく、**大規模データを圧縮した予測器**として理解する。  
3. 生成AIの能力を、**流暢さ・正確さ・有用性・安全性・コスト**などの複数軸に分けて理解する。

> ひとことで言うと、生成AIは  
> **「大規模データから世界の規則性を圧縮した予測器を、事後学習でアシスタント化し、検索・ツール・追加推論時間で現実世界へ接続したシステム」**  
> である。

## 1. エグゼクティブサマリー

### 1.1 生成AIの本質
生成AIの中核は、Transformer 系アーキテクチャを用いた**次トークン予測**である [1][2]。  
この学習は「真実」を直接最適化しているのではなく、「与えられた文脈の続きをもっともらしく予測する」ことを最適化している。そのため、強い汎用性が得られる一方、**流暢さと正しさがズレうる** [2][18]。

### 1.2 能力は一枚岩ではない
生成AIの能力は、少なくとも以下の5層に分けて考えると整理しやすい。

- **基礎能力**: 事前学習で獲得する言語・知識・パターン処理能力 [2][3][4]
- **振る舞い**: 指示追従、有用性、安全性、文体など。事後学習で強く変わる [6][7][8]
- **外部接続**: 検索、DB、API、ツール利用による知識の更新性と実行能力 [11][12][13]
- **推論時の粘り**: chain-of-thought、自己整合性、反復サンプリング、budget forcing など [9][10][24][25]
- **信頼性**: truthfulness、robustness、fairness、toxicity、efficiency など [18][19][20][21]

### 1.3 2024–2025年の重要変化
最近の大きな変化は、能力向上の軸が**訓練時計算だけでなく、推論時計算にも広がった**ことである [24][25][26]。  
これは「考えるモデル」という表現で宣伝されがちだが、より中立に言えば、**test-time compute を増やすことで問題解決能力を押し上げる研究潮流**である。

### 1.4 実務上の帰結
実務で強い生成AIは、たいてい次の構成を持つ。

1. 強い基盤モデル
2. 適切な事後学習
3. 検索または知識ベース接続
4. ツール利用
5. 評価・監視・検証の仕組み

つまり、**強いシステムは「賢い一発回答機」ではなく、「複数のLM呼び出しと外部コンポーネントからなる計算グラフ」**として設計される [11][12][13][26]。

## 2. 生成AIを理解するための7つのメンタルモデル

### 2.1 圧縮された予測器
LLM は、インターネット上の膨大なテキスト・コード・対話データから、世界の規則性を**圧縮**して持っている [2]。  
したがって、LLM は厳密な意味での「知識ベース」ではなく、**圧縮された統計構造から続きを再構成する装置**である。

**含意**
- 高い汎用性が出る
- 曖昧な知識も扱える
- しかし、出力は再構成であるため誤りうる

### 2.2 コンテキスト上で走るプログラム
プロンプト、会話履歴、添付文書、例示は、すべてモデルにとっての**一時的な実行環境**である [2][14][15]。  
そのため、プロンプトは「お願い文」ではなく、**推論時にタスクを構成するプログラム的入力**として考えると理解しやすい。

### 2.3 二段ロケット
生成AIは、大きく二段階で作られる。

- **事前学習**: 次トークン予測で基礎能力を作る [2][3][4]
- **事後学習**: 指示追従・安全性・有用性を整える [6][7][8]

このため、**ベースモデルが賢いこと**と、**アシスタントとして使いやすいこと**は別問題である [6]。

### 2.4 知識は二層にある
知識には少なくとも二つの層がある。

- **パラメトリック知識**: モデルの重みに埋め込まれた圧縮知識 [2]
- **非パラメトリック知識**: 検索、文書、DB、API のような外部知識 [11]

この二層を組み合わせるのが RAG の発想であり、実務ではほぼ必須である [11]。

### 2.5 計算予算は訓練時だけではない
能力は、訓練時の巨大計算だけで決まるわけではない。  
推論時に複数サンプルを生成したり、考える長さを伸ばしたり、検証器を使ったりすることで、性能はかなり変わる [9][10][24][25][26]。

### 2.6 モデルよりシステム
近年の進歩は、単純なモデル性能の向上だけでなく、**RAG、ReAct、Toolformer、compound systems** のようなシステム設計に強く依存している [11][12][13][26]。

### 2.7 流暢さと正しさは別物
LLM はしばしば非常に流暢だが、それは truthfulness を保証しない [18][19][21]。  
このズレを見抜くために、評価は accuracy だけでは足りず、calibration、robustness、fairness、toxicity、efficiency などの多面的指標が必要になる [19]。

## 4. 第I部: LLMの内部

### 4.1 トークン化: モデルは最初から「意味」を見ていない
LLM の入力は自然言語そのものではなく、**トークン列**である [1]。  
モデルは最初から「文」や「概念」を直接扱っているのではなく、まず記号列を扱い、それを埋め込みベクトルへ変換する。

**重要な含意**
- トークン化は単なる前処理ではない
- どこで区切るかが系列長と計算コストを変える
- 同じ人間可読文でも、モデルには別の難しさで見えることがある

### 4.2 自己注意: 配線を動的に作る
Transformer の中核は自己注意である [1]。  
各位置は query / key / value を通じて、**系列中のどこを見るべきか**を動的に決める。  
この仕組みによって、遠距離の依存関係も一段で参照できる。

> 便利な理解:  
> **attention = 位置どうしの情報配線**

### 4.3 FFN: パターン辞書としての記憶
Transformer の feed-forward 層は単なる補助ではなく、モデル全体の多数のパラメータを占める [1]。  
Geva らは、FFN を **key-value memory** として解釈し、各 key が入力パターンに反応し、各 value が出力語彙分布へ影響することを示した [14]。

> 便利な理解:  
> **FFN = パターン辞書 / 局所記憶**

### 4.4 残差接続: 作業台としての表現空間
各層の出力は残差接続を通じて蓄積される [1][14]。  
そのため、LLM は層ごとにすべてを書き換えるのではなく、**既存の表現に少しずつ加工を加える**と考えると理解しやすい。

### 4.5 学習目標: 次トークン予測
GPT-3 の重要性は、175B パラメータという規模だけでなく、**非常に単純な目的関数（次トークン予測）を大規模化することで多様な能力が立ち上がる**ことを示した点にある [2]。

このことは二つの事実を意味する。

1. 多様な能力は、しばしば**副産物として出現**する  
2. しかし、その能力は「真理」ではなく**予測の上手さ**に由来する

### 4.6 In-Context Learning は何なのか
GPT-3 以降、追加学習なしでプロンプト中の例からタスクを切り替える ICL が注目された [2]。  
ただし、そのメカニズムは単純ではない。

- Min らは、正しいラベルをデモに入れなくても性能低下が小さい場合があると示し、**ラベルそのものより、ラベル空間・形式・入力分布が効いている**と論じた [15]
- Pan らは ICL を **task recognition** と **task learning** に分け、前者は比較的小さいモデルでも出るが、後者はスケールと例数により強まることを示した [27]
- Olsson らは **induction heads** が ICL の主要メカニズム候補だと示した [28]
- Akyürek らは単純な線形回帰設定で、Transformer が内部で既知の学習アルゴリズムに似たことをしている可能性を示した [29]

> 現時点の誠実なまとめ:  
> **ICL は、タスク認識・パターン再利用・文脈内での適応的計算が重なって現れる現象であり、単一原理に完全還元されたわけではない。**

### 4.7 スケーリング: なぜ大きくすると賢くなるのか
Kaplan らは、モデルサイズ・データサイズ・計算量に対して性能がべき乗則で改善することを示した [3]。  
しかし Hoffmann らは、固定計算予算の下では**モデルサイズと学習トークン数の釣り合い**が重要だと示した [4]。  
つまり、効くのは「巨大さ」そのものではなく、**適切なスケーリング設計**である。

### 4.8 疎なスケーリング: MoE
Switch Transformer は、すべての入力に同じパラメータを使う dense モデルとは別に、**入力ごとに使う専門家を切り替える疎な Mixture-of-Experts** を示した [5]。  
これは「総パラメータ数」と「一回の推論コスト」を分離する重要な発想である。

### 4.9 長文脈の限界
長いコンテキスト窓があっても、モデルがその中の情報を均等にうまく使えるとは限らない。  
Lost in the Middle は、関連情報が文脈の中央にあると性能が落ちやすいことを示した [21]。

**含意**
- 長文を丸ごと突っ込めばよいわけではない
- 検索して再配置するほうが強い場合がある
- 「コンテキスト長」と「文脈利用能力」は別物

## 5. 第II部: 事後学習・アラインメント

### 5.1 ベースモデルとアシスタントは別物
InstructGPT は、**1.3B の事後調整済みモデルが 175B のベース GPT-3 より人間評価で好まれうる**ことを示した [6]。  
この結果は、生成AIの理解において極めて重要である。

> **能力の多くは事前学習で作られ、振る舞いの多くは事後学習で整えられる。**

### 5.2 RLHF の役割
RLHF（reinforcement learning from human feedback）は、  
人間の好みや意図に沿うようにモデルをチューニングする枠組みである [6]。  
目的は「正解率の最大化」だけではなく、**helpful / honest / harmless** のような人間中心の基準に合わせることにある。

### 5.3 Constitutional AI
Constitutional AI は、ルールや原則の文書を使い、AI 自身の自己批評と改訂を利用して安全性を高める方向を示した [7]。  
これは、人手ラベルに全面依存しない安全性調整として重要である。

### 5.4 DPO
DPO は、RLHF の最終目標を、より単純な分類損失で直接最適化できる形に整理した [8]。  
ここで重要なのは、「好み学習の本質」を理解し直し、**より扱いやすく、安定した事後学習法**を与えた点である。

### 5.5 この章のメンタルモデル
- 事前学習は「賢さ」の土台
- 事後学習は「人にとっての使いやすさ」を作る
- よって、モデルを評価するときは**基礎能力**と**挙動**を分けて見る必要がある

## 6. 第III部: 検索・ツール・エージェント

### 6.1 なぜ外部記憶が必要か
パラメトリック知識は汎用性が高い一方、更新性と出典提示に弱い。  
RAG は、外部文書検索を組み合わせることでこの弱点を補う [11]。

> **モデル内知識 = 圧縮された曖昧な記憶**  
> **検索知識 = 明示的で更新可能な記憶**

### 6.2 RAG
RAG は、検索器で関連文書を取り、その文書を条件に生成する枠組みである [11]。  
実務では、FAQ、社内文書、仕様書、法務文書、製品DB などとの接続で非常に重要である。

### 6.3 ReAct
ReAct は、reasoning と acting を交互に回すことで、LM が外部世界とやりとりしながら問題解決できることを示した [12]。  
ここでの本質は、**思考と行動を分離せず、交互に更新する**ことである。

### 6.4 Toolformer
Toolformer は、LM が「いつ API を呼ぶべきか」を学習できることを示した [13]。  
これは、ツール使用をハードコードではなく、**モデルの学習対象に組み込む**方向性として重要である。

### 6.5 Compound AI Systems
2024年には、複数回の LM 呼び出しと集約を含む **compound inference systems** のスケーリング特性が研究された [26]。  
ここでは、単純な多数決やフィルタリングでも、LM 呼び出し数の増加が性能を改善しうる一方、改善は単調でないことも示された [26]。

### 6.6 この章のメンタルモデル
- モデルは脳みそ
- 検索は記憶
- ツールは手足
- 検証器は現実との接点

強い生成AIは、単なる「一回の出力」ではなく、**retrieve → plan → act → verify** のループとして設計される。

## 7. 第IV部: 画像生成とマルチモーダル

### 7.1 画像生成は自己回帰とは別の流儀
テキスト生成の主流が自己回帰であるのに対し、画像生成の主流は diffusion である [16][17]。  
DDPM は、ノイズを少しずつ取り除く逆過程として画像生成を定式化した [16]。

> 便利な理解:  
> **画像生成は「描く」より、「ノイズを意味ある構造へ冷却する」に近い。**

### 7.2 Latent Diffusion
Latent Diffusion は、画像ピクセル空間ではなく潜在空間で diffusion を行い、  
高画質と計算効率を両立した [17]。  
現在の画像生成システム理解の基礎として非常に重要である。

### 7.3 CLIP
CLIP は、画像とテキストの対応予測を通じて、自然言語で視覚概念を参照できる強い表現を学習した [22]。  
これにより、「犬」「油絵風」「宇宙服を着た猫」などの概念を自然言語で制御しやすくなった。

### 7.4 Flamingo
Flamingo は、強い視覚モデルと強い言語モデルを橋渡しし、画像や動画を交えた few-shot 学習を可能にした [23]。  
ここで、言語モデルと視覚モデルは別々の島ではなくなり、**共通の意味空間を共有する方向**が明確になった。

### 7.5 この章のメンタルモデル
- テキスト生成の主流は自己回帰
- 画像生成の主流は diffusion
- マルチモーダル化は「複数の感覚入力を共通の意味空間に押し込む」試み

## 8. 第V部: 評価・安全性・社会的視点

### 8.1 TruthfulQA
TruthfulQA は、モデルが人間の誤信念や通俗的な誤りをどれだけ模倣するかを測る [18]。  
ここで重要なのは、**大きいモデルが必ずしもより truthful ではない**ことを示した点である。

### 8.2 HELM
HELM は、言語モデル評価を accuracy 一軸から解放し、  
calibration、robustness、fairness、bias、toxicity、efficiency などを含む多面的評価を提示した [19]。

### 8.3 Stochastic Parrots
Bender らの “Stochastic Parrots” は、巨大言語モデルを、  
データの偏り、文書化負債、環境コスト、権力構造を含む**社会技術システム**として捉えるべきだと警告した [20]。

### 8.4 Lost in the Middle
長文脈モデルの能力に対して、「長い入力を入れられる」と「長い入力をうまく使える」を分けて考える必要を示した [21]。

### 8.5 この章のメンタルモデル
能力評価は少なくとも以下の4軸に分けるべきである。

1. **能力**: タスクを解けるか
2. **挙動**: 指示に従い、役に立つか
3. **信頼性**: 事実性、頑健性、較正、再現性はあるか
4. **外部性**: 偏り、エネルギー、ガバナンスへの影響はどうか

## 9. 第VI部: 2024–2025年の変化 ― 推論時計算の再発見

### 9.1 Chain-of-Thought と Self-Consistency
CoT は、中間推論ステップを書かせることで難しい問題の性能が伸びることを示した [9]。  
Self-Consistency は、単一の思考経路に賭けるのではなく、複数経路をサンプルして整合する答えを選ぶと性能が上がることを示した [10]。

### 9.2 Repeated Sampling
Large Language Monkeys は、反復サンプリングによって「どれか一つが解ける」確率が大きく伸びることを示し、  
推論時計算にもスケーリング則のような挙動があることを示した [24]。

### 9.3 s1
s1 は、小規模な高品質データと **budget forcing** によって、  
推論時に考える長さを制御し、性能を伸ばせることを示した [25]。  
ここで重要なのは、派手な新アーキテクチャでなくても、**推論時の手続き設計そのものが能力を左右する**と示した点である。

### 9.4 DeepSeek-R1
DeepSeek-R1 は、強化学習を中心に reasoning 能力を押し上げる流れを前景化した [26]。  
特に、純粋な RL による reasoning 挙動の出現、そこからの multi-stage 改良、蒸留による小型モデル展開は、近年の reasoning 研究を象徴している [26]。

### 9.5 この章のメンタルモデル
- 事前学習で能力の土台を作る
- 事後学習で振る舞いを整える
- 推論時に追加計算で能力を押し上げる

つまり、知能は**訓練済み重みの中だけ**に閉じていない。  
実際の性能は、**どれだけ考えさせ、どれだけ検証させるか**にも依存する。

## 10. 実務への含意

### 10.1 プロンプトよりコンテキスト設計
「良いプロンプトを書けば何とかなる」という考えは不十分である。  
効くのは、例示、形式、情報の配置、検索、検証まで含む**コンテキスト設計**である [2][15][21]。

### 10.2 モデル選定は目的別に
- 長文要約や対話の自然さ
- 数学やコード推論
- 検索前提のQA
- 画像生成や視覚QA

これらは同じ強さではない。  
**用途に対して何を最適化したモデル・システムか**を見る必要がある。

### 10.3 RAG は小手先ではない
最新性、社内固有知識、出典提示、監査可能性が必要なとき、RAG は「補助」ではなく**本体の一部**である [11][21]。

### 10.4 評価を最初に設計する
生成AI開発では、モデル導入の前に**何を成功と呼ぶか**を決めるべきである。  
HELM 的な発想で、accuracy 以外の評価軸も早めに決めることが重要である [19]。

### 10.5 ツールと検証器を持たせる
計算、検索、実行、照合、ルール検査のようなツールは、  
モデルの弱点を補う。  
LM 単体の能力を信じ切るより、**弱点をシステムで包む**ほうが堅い [12][13][26]。

## 11. よくある誤解

### 誤解1: 生成AIは知識データベースである
違う。  
生成AIは、知識を厳密に格納したデータベースではなく、**圧縮された予測モデル**である [2][18]。

### 誤解2: モデルが大きければ必ずよい
違う。  
Chinchilla が示したように、サイズ・データ・計算の配分が重要である [4]。

### 誤解3: 長文脈モデルなら全部読める
違う。  
読めることと、うまく使えることは別である [21]。

### 誤解4: RLHF をすれば安全で正確になる
違う。  
RLHF は挙動改善に有効だが、truthfulness や robustness を自動で保証しない [6][18][19]。

### 誤解5: reasoning はモデル固有能力だけで決まる
違う。  
推論時のサンプリング、自己整合、検証、budget forcing などでも大きく変わる [10][24][25][26]。

### 誤解6: 画像生成もLLMと同じ仕組みで動く
違う。  
画像生成の主流は diffusion であり、自己回帰テキスト生成とは設計思想が異なる [16][17]。

## 12. 補足しておくと理解が深まる論点

ここまでの議論に加えて、今後学ぶと理解がさらに深まる論点を挙げる。

### 12.1 データ品質とデータカリキュラム
モデル性能はアーキテクチャだけでなく、**どのデータを、どの順で、どれだけ見せるか**に大きく依存する。  
Chinchilla 以後は特に、データ量だけでなくデータ品質の議論が重要になっている [4]。

### 12.2 合成データと蒸留
近年の reasoning モデルや小型モデルの多くは、  
合成データや蒸留を強く活用している [26]。  
これは「教師なし事前学習だけではない」現代的なモデル開発の特徴である。

### 12.3 ベンチマーク汚染
大規模 web 学習では、評価データが訓練データへ混入する問題が常にある。  
そのため、ベンチマーク性能は**能力そのもの**ではなく、  
データ由来の要因も含む可能性がある。

### 12.4 監査可能性
実運用では、出力の良し悪し以上に、  
**なぜその答えになったかを追跡できるか**が重要になる。  
この観点でも、RAG やツール利用、ログ設計は重要である [11][12][13]。

### 12.5 推論コストと製品設計
推論時スケーリングは性能を伸ばすが、  
同時に遅延・コスト・運用複雑性を増やす。  
したがって、性能だけでなく**どこに計算を配分するか**が製品設計の核心になる [24][25][26]。

## 14. 用語集

**Transformer**  
自己注意機構を中核とするニューラルネットワークアーキテクチャ。現代 LLM の基盤 [1]。

**Self-Attention**  
系列中の各位置が、他の位置をどれだけ参照すべきかを動的に決める機構 [1]。

**In-Context Learning (ICL)**  
重み更新なしに、プロンプト中の例からタスクを切り替える現象 [2]。

**RLHF**  
人間のフィードバックを報酬信号として使い、モデルの振る舞いを調整する方法 [6]。

**RAG**  
検索で取得した文書を文脈に加えて生成する設計 [11]。

**Diffusion Model**  
ノイズ除去過程としてデータ生成をモデル化する生成手法 [16]。

**CLIP**  
画像とテキストの対応予測を通じて共通表現を学ぶモデル [22]。

**Truthfulness**  
答えが真実に近いかどうか。流暢さとは別軸 [18]。

**Test-time scaling**  
推論時の追加計算（複数サンプル、長い思考、検証など）で性能を高める考え方 [24][25][26]。

**Compound AI System**  
単一のLM呼び出しではなく、複数回のLM呼び出し、検索、ツール、投票、検証器などで構成されたシステム [26]。

## 15. 最終まとめ

生成AIを理解するうえで最も大事なのは、  
それを**「すごく賢い会話ソフト」**としてではなく、  
**「大規模データを圧縮した予測器を、事後学習・検索・ツール・追加推論計算で拡張したシステム」**として見ることである。

この見方を採用すると、個々の研究の位置づけが整理できる。

- Transformer は**土台**を作った [1]
- GPT-3 は**大規模化と ICL** の意味を可視化した [2]
- Scaling Laws と Chinchilla は**どのように大きくするか**を教えた [3][4]
- InstructGPT、Constitutional AI、DPO は**使いやすいアシスタント化**を押し進めた [6][7][8]
- RAG、ReAct、Toolformer は**外部世界との接続**を与えた [11][12][13]
- DDPM、LDM、CLIP、Flamingo は**画像・視覚・マルチモーダル**へ広げた [16][17][22][23]
- TruthfulQA、HELM、Stochastic Parrots、Lost in the Middle は**限界と評価軸**を明確化した [18][19][20][21]
- Large Language Monkeys、s1、DeepSeek-R1、compound systems は**推論時スケーリングとシステム設計**を次の主戦場にした [24][25][26]

したがって、新しい論文や製品を見たときは、まず次の問いで仕分けするとよい。

1. これは**アーキテクチャ**の進歩か
2. これは**スケーリング**の進歩か
3. これは**事後学習・アラインメント**の進歩か
4. これは**検索・ツール・エージェント化**の進歩か
5. これは**マルチモーダル化**の進歩か
6. これは**推論時計算**の進歩か
7. これは**評価・安全性**の進歩か

この仕分けができるようになると、生成AIの世界で次々に現れる流行語に振り回されにくくなる。  
霧は完全には晴れないが、少なくとも「どの山を見ているのか」は分かるようになる。
