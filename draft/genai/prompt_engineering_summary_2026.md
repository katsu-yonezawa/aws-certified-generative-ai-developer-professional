# プロンプトエンジニアリングの現在地と、いま本当に重要な考え方

## 1. この文書の要旨

結論から言うと、**プロンプトエンジニアリングは今でも重要**です。
ただし、以前のような「魔法の言い回し」や「裏技」としての重要性は相対的に下がり、現在は **AIに何を・どこまで・どの形式でやらせるかを明確に定義する基礎技術** という位置づけになっています。

そして実務上は、プロンプトそのものよりも、次の考え方のほうが重要度が高くなっています。

1. **成功条件の定義**
2. **評価（evals）**
3. **コンテキスト設計（context engineering）**
4. **ツール利用・構造化出力**
5. **その上でのプロンプト改善**
6. **必要に応じた fine-tuning**

一言でまとめると、

> **プロンプトエンジニアリングは今も重要。ただし、より重要なのは「モデルが何を知り、何ができ、どう評価されるか」を設計すること。**

---

## 2. いまでもプロンプトエンジニアリングは重要か

重要です。ただし意味が変わりました。

昔の議論では、プロンプトエンジニアリングは「少し表現を変えるだけで劇的に性能が変わるテクニック」として語られがちでした。これは初期のLLMではかなり当たっていた面があります。しかし最新世代では、モデル自体の指示追従性能が高くなり、**凝った言い回しよりも、目的・前提・制約・出力形式を明確に示すこと**のほうが効きやすくなっています。

つまり、いまのプロンプトエンジニアリングは次のようなものです。

- ふわっと頼まない
- 何を出力すれば成功かを明確にする
- 不明な点は「不明」と書かせる
- 出力形式を指定する
- 必要なら例を見せる
- 必要な資料や前提を与える

呪文ではなく、**仕様書の簡易版**に近いと考えるとしっくりきます。

---

## 3. いま、プロンプトより重要な考え方

### 3-1. 成功条件の定義

AI活用で最初に必要なのは、「良い出力」とは何かを決めることです。

たとえば、

- 正確であればよいのか
- 網羅的であればよいのか
- 短く読みやすければよいのか
- 出典が必要なのか
- 断定を避けるべきなのか
- 表形式である必要があるのか

この条件が曖昧だと、プロンプト改善も評価もできません。

### 3-2. 評価（evals）

AIは毎回少しずつ出力がぶれます。したがって「なんとなく良くなった気がする」で運用すると、すぐに品質管理が崩れます。

そのため、重要なのは **evals** です。つまり、

- 何をテストするか
- どんな入力を使うか
- どう採点するか
- 変更前後で何を比較するか

をあらかじめ決めておくことです。

これは実務ではかなり本丸です。プロンプト改善も、モデル変更も、RAG改善も、最終的には評価なしでは正しく比較できません。

### 3-3. コンテキスト設計（context engineering）

現代のLLM活用では、「どう書くか」より「何を渡すか」が非常に重要です。

ここでいうコンテキストとは、単なる入力文だけではありません。たとえば以下を含みます。

- system instructions
- 会話履歴
- 添付資料
- 検索結果
- 外部データ
- 利用可能なツール
- 出力フォーマットの定義

AIが賢く見えるかどうかは、モデルの性能だけでなく、**必要な情報と道具を適切に渡せているか**でかなり決まります。

### 3-4. ツール利用・構造化出力

自由文だけでAIを使うと、出力のぶれが大きくなります。
そこで重要になるのが次の2つです。

- **構造化出力**：JSONや決まった表形式など、出力形式を固定する
- **ツール利用**：検索、DB参照、計算、外部API呼び出しなどをAIに使わせる

「有効なJSONで返してね」とお願いするだけより、**スキーマで縛る**ほうが安定します。ここが、単なるチャット利用と、実務システムとしての利用の分かれ目です。

### 3-5. モデル選定

失敗を全部プロンプトの問題だと考えるのは危険です。

- 複雑な意思決定に向くモデル
- 高速・低コストな実行に向くモデル
- ツール利用に強いモデル
- 長いコンテキストに強いモデル

は同じではありません。

つまり、「プロンプトを工夫する前に、そもそもその仕事に向いたモデルか」を考える必要があります。

### 3-6. fine-tuning は最後

fine-tuning は強力ですが、最初に飛びつくものではありません。

先にやるべきは、

1. 成功条件の定義
2. 評価の設計
3. コンテキストの整理
4. プロンプト改善

です。

それでも安定して足りない部分が残るときに、繰り返しタスク向けに fine-tuning を検討する、という順番が実務的です。

---

## 4. 境界線：誰がどこまで知っていればよいか

このテーマは、**一般ユーザー / 専門職 / 技術者・AI導入担当** の3層で考えるのが実践的です。

---

## 5. 一般ユーザーはここまで知っていれば十分

一般ユーザーに必要なのは、「うまくお願いする」よりも **依頼を明確にする力** です。

### 知っておくべきこと

- 目的を明確に伝える
- 前提条件を渡す
- 制約を明示する
- 出力形式を指定する
- 不明な点は不明と書かせる
- 必要な資料は貼る
- 形式が大事なときだけ例をつける

### 一般ユーザー向けテンプレート

```text
目的:
この作業で最終的に達成したいことは〇〇です。

前提:
対象読者は〇〇。背景は〇〇。

使用してよい情報:
以下の文章だけを使ってください。外部知識は使わないでください。
[資料を貼る]

制約:
・300字以内
・専門用語は避ける
・箇条書き3点まで

出力形式:
1. 重要点3つ
2. 根拠の引用
3. 結論

不明点の扱い:
資料から言えないことは「不明」と明記してください。
```

### できなくてもよいこと

一般ユーザーは、以下まで知る必要はありません。

- API
- JSON Schema
- function calling
- evals / graders
- prompt caching
- compaction
- fine-tuning

### 到達基準

**1回の会話で、曖昧な依頼を具体的な依頼に変えられること。**

ここまでできれば十分実用的です。

---

## 6. 専門職はここまで知っているべき

ここでいう専門職には、たとえば以下を含みます。

- 法務
- 人事
- 企画
- 研究職
- コンサルタント
- 編集
- マーケティング
- 教育担当

この層では、「たまに当たる」では足りません。必要なのは **再現性** と **根拠** です。

### 知っておくべきこと

- プロンプトを毎回ゼロから書かず、テンプレ化する
- 出力に求める評価観点を先に決める
- 一発で全部やらせず、抽出→分類→要約→提案のように分解する
- 根拠を先に引用させ、その後で判断させる
- 不明点は断定させない
- 少数の代表ケースで比較する
- 人間のレビュー観点を固定する

### 専門職向けの典型テンプレート

```text
あなたの役割:
企業法務アシスタント。曖昧な断定は禁止。

手順:
1. まず文書中の関連箇所を引用
2. 次に論点を分類
3. 最後に、文書から言えること / 言えないことを分けて結論化

評価基準:
・引用と結論が対応している
・推測を事実として書かない
・不明点を不明と書く

出力形式:
表形式
列 = 論点 / 根拠引用 / 判断 / 確信度
```

### 到達基準

**同じ仕事を別の日にやっても、品質が大きくぶれないこと。**

この層では、プロンプトの小技よりも **業務フローの設計** が主役になります。

---

## 7. 技術者・AI導入担当はここまで知っているべき

この層は、プロンプトを「文章」ではなく **契約** として扱う必要があります。

### 必須で押さえるべきこと

#### 7-1. モデル選定

- 複雑な推論に向くモデル
- 高速・低コスト処理に向くモデル
- 長文コンテキストに強いモデル
- ツール利用に強いモデル

をタスクに応じて選ぶこと。

#### 7-2. 構造化出力

- JSON Schema
- 表形式の厳格化
- フィールド単位の検証
- サーバ側バリデーション

自由文に頼りすぎず、**機械処理しやすい形で出させる**ことが重要です。

#### 7-3. ツール呼び出し

- 検索
- DBアクセス
- 外部API
- 計算
- 社内ナレッジ参照

などをモデルに使わせる設計を理解すること。

#### 7-4. evals と graders

最低限、以下を設計できるべきです。

- どんな入力セットで評価するか
- 合格条件をどう定義するか
- 部分点をどう扱うか
- プロンプト変更・モデル変更・RAG変更をどう比較するか

#### 7-5. コンテキスト管理

- system prompt の設計
- 会話履歴の整理
- RAG で渡す文書の選定
- long context の使い方
- caching
- compaction
- 大量ツールの遅延ロード

などを理解し、**コスト・品質・遅延** のトレードオフを設計できること。

#### 7-6. セキュリティとガードレール

- prompt injection
- unsafe commands
- 権限制御
- サーバ側検証
- ログと監査

モデルに好き勝手やらせないための門番が必要です。

#### 7-7. fine-tuning をやる条件

- 繰り返し頻度が高い
- タスクが十分に定義されている
- 評価基準がある
- prompt / context 最適化後も改善余地がある

こうした条件が揃ったときに検討するのが筋です。

### 到達基準

**変更のたびに測定し、品質を保ったまま改善できること。**

技術者の仕事は、たまたま良い出力を得ることではなく、**壊れにくい仕組みを作ること**です。

---

## 8. few-shot はどう扱うべきか

few-shot は今でも有効です。ただし万能ではありません。

### どう考えるべきか

- 形式が重要なときは有効
- 分類や抽出の基準を揃えたいときに有効
- ただし、最新の推論モデルでは zero-shot で十分なことも多い
- few-shot の有無は、最終的には eval で決めるべき

### 教える境界線

- **一般ユーザー**：形式が大事なら例をつける
- **専門職**：評価観点を満たす例を設計する
- **技術者**：few-shot の有無も含めて実験・比較する

つまり、「few-shot は常に正義」でも「few-shot はもう古い」でもありません。タスクとモデルに応じて使い分けます。

---

## 9. 実践的な講座に落とし込むならどうするか

講座として設計するなら、プロンプトの小技から始めて、最終的に **評価・コンテキスト・ツール設計** に着地させるのが実務向きです。

### 推奨構成

#### 第1回：依頼の書き方

- 目的
- 前提
- 制約
- 出力形式
- 不明時の扱い

#### 第2回：根拠ベース運用

- 引用先出し
- 資料限定
- 出典付き
- 「わからない」を許可

#### 第3回：専門職向けテンプレ化

- 用途別テンプレート
- レビュー観点の固定
- タスク分解

#### 第4回：評価の考え方

- 良い出力の定義
- 少数ケース比較
- 改善前後の確認

#### 第5回：技術者向け基礎

- structured outputs
- function calling
- evals
- graders

#### 第6回：context engineering

- RAG
- long context
- caching
- compaction
- tool search

### この講座構成の意図

こうすると、「プロンプトのコツ」を入り口にしつつ、受講者を現代の本流である **設計・評価・運用** へ自然に導けます。

---

## 10. 一言で境界を切るなら

- **一般ユーザー** は「上手に頼める」まで
- **専門職** は「根拠付きで再現できる」まで
- **技術者・AI導入担当** は「測って改善できる」まで

この切り方が、現時点ではかなり実務的です。

---

## 11. 実務でそのまま使えるチェックリスト

### 一般ユーザー向けチェックリスト

- 目的を書いたか
- 読者や前提を書いたか
- 長さや形式を指定したか
- 使ってよい資料を渡したか
- 不明な点は不明と書かせているか

### 専門職向けチェックリスト

- 根拠を先に抽出させているか
- 評価観点が定義されているか
- タスクを段階に分けているか
- 代表ケースで比較しているか
- レビュー基準が固定されているか

### 技術者向けチェックリスト

- モデル選定の理由があるか
- 出力形式を検証できるか
- ツール利用を安全に制御できるか
- eval セットがあるか
- 回帰を検知できるか
- コンテキストの肥大化を制御できるか
- セキュリティ対策があるか

---

## 12. 最終結論

プロンプトエンジニアリングは、いまでも消えていません。
ただし、その役割は **「上手な言い回し探し」から「AIへの仕事の定義」へ** 変わりました。

そして、より重要になったのは次の力です。

- 成功条件を定義する力
- 評価する力
- 必要な情報とツールを渡す力
- 出力を構造化する力
- モデルや運用全体を設計する力

したがって、これからのAI活用では、

> **プロンプトのコツを知っている人** より、
> **AIの仕事の与え方・測り方・仕組み化のしかたを知っている人** のほうが強い。

これが、ここまでの議論全体の結論です。
