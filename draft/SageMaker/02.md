試験ガイドでは、SageMaker AI、SageMaker Clarify、Data Wrangler、Ground Truth、JumpStart、Model Monitor、Model Registry、Neo、Processing、Unified Studio などが対象範囲に含まれています。加えて、programmatic model cards、data lineage、drift、bias drift monitoring なども明示されています。

## 0. 先に全体像をつかむ

SageMaker AI の学習は、次の流れで見ると理解しやすくなります。

1. データを整える  
   例: CSV、ログ、画像、音声、JSON を学習・推論に使いやすい形へそろえる

2. 必要ならラベルを付ける  
   例: 問い合わせ履歴に 苦情 / 質問 / 要望 の正解ラベルを付与する

3. モデルを選ぶ・調整する  
   例: JumpStart からモデルを選び、社内ドメイン向けに微調整する

4. 本番へ配備する  
   例: チャットならリアルタイム、夜間一括処理ならバッチを選ぶ

5. 品質や安全性を評価する  
   例: バイアス、説明可能性、ドリフト、モデル品質を確認する

6. 承認・監査・再現性を整える  
   例: どのデータとどの学習ジョブから、そのモデルができたかを追跡する

この流れで各機能を見ると、用語がばらばらに見えにくくなります。

## 1. まず最重要: Bedrock と SageMaker AI の境目

### 1-1. 一言でいうと

- **Amazon Bedrock** は、既存の基盤モデルを API で素早く使い、生成 AI アプリを組み立てる方向に強い
- **Amazon SageMaker AI** は、モデル構築・学習・微調整・配備・監視・MLOps まで含めて、深く制御したいときに強い

この2つは競合というより、**役割が違う兄弟** のようなものです。  
試験でも、どちらが優れているかではなく、**要件に対してどちらが自然か** を見極める問題として出やすいです。

### 1-2. よくある実務イメージ

#### Bedrock が自然な例

社内ナレッジを参照して回答するチャットボットを短期間で作りたい。  
モデルは Claude や Nova など既存 FM を使い、インフラ管理はあまりしたくない。  
RAG、Guardrails、Agents、Model Evaluation などを組み合わせて、まず動くものを素早く出したい。

#### SageMaker AI が自然な例

既存モデルでは足りず、社内固有の専門用語や帳票形式に合わせて、データ前処理から微調整、推論方式の選定、本番監視、承認フローまでしっかり組みたい。  
モデルの版管理、評価結果、ドリフト監視、承認済みモデルだけの本番投入まで求められる。

## 2. SageMaker AI と Bedrock の比較表

| 観点               | Amazon Bedrock                                                   | Amazon SageMaker AI                                                 |
| ------------------ | ---------------------------------------------------------------- | ------------------------------------------------------------------- |
| 主な役割           | 既存 FM を使って生成 AI アプリを素早く作る                       | モデル構築、学習、微調整、配備、監視を広く制御する                  |
| 向いている人       | アプリ開発者、GenAI 利用を素早く進めたいチーム                   | ML エンジニア、データサイエンティスト、深く制御したい開発者         |
| 典型ユースケース   | チャットボット、要約、RAG、エージェント、画像生成アプリ          | カスタムモデル、専門領域への微調整、独自推論基盤、MLOps             |
| モデル利用の考え方 | 基本はマネージドな既存 FM を API で利用                          | モデル選定から学習、配備まで幅広く制御                              |
| カスタマイズ       | 一部 FM の fine-tuning、pre-training、custom model import に対応 | 前処理、学習、微調整、ハイパーパラメータ、配備方式まで広く制御      |
| インフラ管理       | サーバーレス寄りで、利用開始が軽い                               | エンドポイントや学習ジョブなど、より細かな設計が必要                |
| 配備の考え方       | API 利用が中心                                                   | Real-time、Serverless、Async、Batch などを要件に応じて選ぶ          |
| 監視・運用         | GenAI 機能寄りの運用                                             | Model Monitor、Model Registry、Pipelines、Lineage など MLOps が厚い |
| ガバナンス         | Guardrails、Model Evaluation、利用ログなど                       | Model Cards、Model Registry、Lineage、継続監視など                  |
| コスト感の考え方   | API 呼び出しやトークン利用中心で考えやすい                       | 学習・推論の計算資源、ストレージ、監視コストも考える                |
| 試験での見分け方   | 既存 FM を速く安全に使いたい                                     | 独自調整、学習、承認、監視、本番運用までしっかり回したい            |

### 2-1. 試験向けの見分け方

問題文に次のような言葉が見えたら、まず Bedrock を疑います。

- 既存の基盤モデルを API で使う
- RAG をすばやく作る
- Guardrails で安全性制御したい
- Agents や Prompt Management を使う
- インフラ管理を最小限にしたい

逆に、次のような言葉が見えたら SageMaker AI の可能性が高いです。

- 学習、微調整、ハイパーパラメータ調整
- 独自コンテナ、独自フレームワーク
- 推論方式の選定
- ドリフト監視、モデル版管理、承認フロー
- MLOps、再現性、lineage、model cards

### 2-2. 試験で引っかかりやすい点

**Bedrock でも一部カスタマイズはできる** ので、Bedrock は全く調整できない、と覚えるのは不正確です。  
ただし、**機械学習ワークフロー全体を深く制御するなら SageMaker AI の方が自然** です。

つまり、次のように整理すると落ち着きます。

- Bedrock: 既存 FM を中心に、アプリ開発を速く進める
- SageMaker AI: 学習と運用の自由度を上げる

## 3. SageMaker AI の機能を流れで整理する

## 3-1. データ準備

### SageMaker Data Wrangler

**役割**  
表形式データを取り込み、整形し、変換し、分析するための機能です。  
探索的にデータを見ながら、前処理パイプラインのたたき台を作るイメージです。

**使う場面**  
たとえば、顧客テーブル、購入履歴、問い合わせ履歴が別々にあり、そのままでは学習用に使いにくい場面です。  
列の欠損補完、不要列削除、カテゴリ変換、結合、簡単な可視化などを進めます。

**試験での見分け方**  
問題が表形式データの前処理や準備の話なら、まず候補になります。  
とくに、学習前の下ごしらえという雰囲気が強いときに自然です。

### SageMaker Processing

**役割**  
前処理、後処理、評価データ作成などを、フルマネージドな処理ジョブとして実行します。

**使う場面**  
たとえば、次のようなケースです。

- 音声ファイルを文字起こしして JSONL に変換する
- 画像とメタデータを学習用フォーマットへまとめる
- 学習後に推論結果を加工して評価用の表を作る

**Data Wrangler との違い**  
Data Wrangler は、対話的に整える入口として理解するとわかりやすいです。  
Processing は、本番ワークフローの中で定期実行できる処理ジョブとして考えると整理しやすいです。

### SageMaker Ground Truth

**役割**  
人手を使ってラベル付きデータを作るサービスです。

**使う場面**  
たとえば、サポートメールに 障害 / 要望 / 問い合わせ の正解ラベルを付けたいが、教師データがまだない場面です。  
そのとき、まず人がラベルを付け、モデル学習に使える形にします。

**試験での見分け方**  
正解ラベルを人手で作る話が出たら、Ground Truth を疑うとよいです。

## 3-2. モデル選定とカスタマイズ

### SageMaker JumpStart

**役割**  
事前学習済みモデルやサンプルを見つけ、試し、必要なら微調整し、配備する入口です。

**使う場面**  
たとえば、社内 FAQ に強いモデルを作りたいが、完全にゼロから学習するのは重すぎる場面です。  
まず近いモデルを探して、そのモデルを自社向けに調整します。

### Fine-tuning の理解

Fine-tuning は、**すでに学習済みのモデルを、自分の目的に合わせて追加学習すること** です。  
最初から全部学習し直すわけではないので、ゼロから学習するより現実的です。

#### domain adaptation

**意味**  
業界用語、社内文書、専門文体などに寄せたいときの微調整です。

**実例**  
医療、金融、製造、法務のように、一般的な言い回しでは足りず、専門表現に自然に対応してほしいケースです。

#### instruction-based fine-tuning

**意味**  
プロンプトと望ましい応答の組を与えて、答え方やタスク実行の仕方を整える微調整です。

**実例**  
問い合わせ文に対して、次の形式で答えてほしいとします。

- まず結論
- 次に根拠
- 最後に注意点

このような出力様式を安定させたいときに向きます。

### 試験向けの整理

- 専門用語や社内文書の癖に寄せたい → domain adaptation
- 指示に対する答え方や出力フォーマットを寄せたい → instruction-based fine-tuning
- そもそも既存モデルをすぐ試したい → JumpStart から入る

## 3-3. 本番配備と推論

ここはかなり重要です。  
問題文で **レイテンシー、トラフィック特性、処理時間、入力サイズ、コスト** を見て、推論方式を選ぶ問題が出やすいです。

### 基本の考え方

- すぐ答えが必要か
- 使われる頻度は高いか低いか
- 入力や処理は重いか軽いか
- 一件ずつ処理するか、まとめて処理するか

この4点を見るだけでも、だいぶ切り分けできます。

---

## 4. 推論方式の比較表

| 推論方式                             | 向いている場面                                  | レイテンシーの考え方               | 目安となる制約                                                        | 典型例                                       | 強み                                                                  | 注意点                                                     |
| ------------------------------------ | ----------------------------------------------- | ---------------------------------- | --------------------------------------------------------------------- | -------------------------------------------- | --------------------------------------------------------------------- | ---------------------------------------------------------- |
| Real-time Inference                  | 即時応答が必要、継続的なトラフィックがある      | 低レイテンシー重視                 | ペイロード最大 25 MB、通常応答は 60 秒、ストリーミング応答は最大 8 分 | チャット、即時分類、画面上のリアルタイム補助 | 常時応答しやすい、オートスケーリング可能                              | エンドポイントを持ち続けるのでアイドル時もコストが出やすい |
| Serverless Inference                 | 断続的な利用、アクセス予測がしにくい            | 多少のコールドスタートを許容       | ペイロード最大 4 MB、処理時間は最大 60 秒                             | 社内向け要約ツール、たまに使う推論 API       | インフラ管理が軽い、アイドル課金を抑えやすい                          | 低遅延の常時利用には不向きなことがある                     |
| Asynchronous Inference               | 大きな入力、長い処理時間、即時応答が不要        | 非同期で待てる前提                 | ペイロード最大 1 GB、処理時間は最大 1 時間                            | 長文解析、重い画像処理、マルチモーダル処理   | 大きなペイロードや長時間処理に向く、利用がないとき 0 台まで縮退できる | その場で即答する UI には不向き                             |
| Batch Transform                      | まとめて一括推論したい                          | オフライン処理前提                 | 大規模データセットの一括処理向け                                      | 毎晩の一括分類、履歴データ処理               | 常時エンドポイント不要、大量データ向け                                | リアルタイム用途には向かない                               |
| Multi-model endpoint                 | 多数のモデルを 1 エンドポイントで効率運用したい | モデル読込の影響を受けることがある | CPU / GPU バックエンドの複数モデル運用を支援                          | テナント別モデル、店舗別モデル               | コスト効率がよい、モデル数が多いとき有効                              | 低頻度モデルはコールドロードが起きやすい                   |
| Inference pipeline / Multi-container | 前処理 → 推論 → 後処理を 1 本でつなぎたい       | 段数ぶんの遅延を考える             | 複数コンテナを順次呼び出して構成                                      | 整形してから分類、分類後にしきい値判定       | 推論経路を一体化できる                                                | 単純な1モデル推論より構成が複雑                            |

### 4-1. すぐ判断するための覚え方

- **今すぐ答える必要がある** → Real-time
- **たまにしか呼ばれない** → Serverless
- **重くて待てる** → Async
- **大量データをまとめて処理** → Batch Transform
- **モデルがたくさんある** → Multi-model endpoint
- **前後処理もまとめて1本化したい** → Inference pipeline

### 4-2. 実例でイメージする

#### Real-time Inference の例

カスタマーサポート画面で、オペレーターが顧客の質問を入力すると、補助回答候補を 1 秒台で返したい。  
この場合は、待たせると体験が悪くなるので Real-time が自然です。

#### Serverless Inference の例

社内の議事録要約ツールを、数十人が必要なときだけ使う。  
普段はほとんど呼ばれないので、常時エンドポイントを持つと無駄が出やすい。  
この場合は Serverless が似合います。

#### Asynchronous Inference の例

1 回の依頼で長文 PDF を何十ページも解析し、表や画像の説明まで含めてまとめたい。  
すぐ返す必要はなく、数分待てればよい。  
この場合は Async が自然です。

#### Batch Transform の例

毎晩、前日分の問い合わせ 20 万件に対して、カテゴリ分類と感情分析を一括でかける。  
人が待つわけではないので、Batch Transform がわかりやすい選択になります。

#### Multi-model endpoint の例

顧客企業ごとに少しずつ違うモデルを何十本も持っている。  
全部を個別エンドポイントにすると高くつく。  
このとき、Multi-model endpoint が候補になります。

#### Inference pipeline の例

受け取った文章をまず正規化し、そのあと分類モデルへ通し、最後に業務ルールで出力整形して返したい。  
こういう直列処理なら Inference pipeline が考えやすいです。

## 5. 推論方式の試験で出やすい見分けポイント

### Real-time Inference

- 低レイテンシー
- 継続的トラフィック
- REST API 的な常時応答
- ユーザーの操作に即応

### Serverless Inference

- 断続的トラフィック
- アイドル時間が長い
- インフラ管理を軽くしたい
- コールドスタート許容

### Asynchronous Inference

- 大きいペイロード
- 長い処理時間
- リクエストをキューイング
- 即時応答でなくてよい

### Batch Transform

- オフライン一括処理
- 大量データ
- 定期ジョブ
- 常時エンドポイント不要

## 6. 推論関連の補助機能

### Inference Recommender

**役割**  
推論性能とコストの観点で、どの構成が合うかを検討する助けになる機能です。

**使う場面**  
たとえば、同じモデルでもどのインスタンスタイプが適切か迷うときです。  
性能を見ながら、過不足の少ない構成を探す材料になります。

### Multi-model endpoint

**役割**  
多くのモデルを 1 つのエンドポイントで運用するための仕組みです。

**使う場面**  
企業別、地域別、製品ライン別のように、似たモデルを多数持つケースです。

**注意点**  
モデルが常駐していない場合、読み込み待ちが発生しうるため、毎回きわめて低遅延が必要な用途には慎重さが要ります。

### Inference pipeline / Multi-container endpoint

**役割**  
前処理・推論・後処理を 1 エンドポイントで連携させる仕組みです。

**使う場面**  
そのままモデルに入れられない入力を、前段で整形してから推論したい場面です。

## 7. 評価・監視・ガバナンス

## 7-1. SageMaker Clarify

**役割**  
バイアス検出、説明可能性、責任ある AI の観点を扱う機能です。  
本番モデルの bias drift や feature attribution drift の監視にもつながります。

**実例**  
融資審査や採用支援のように、予測結果の偏りや説明責任が気になる場面です。  
また、複数候補モデルの品質や責任の観点を比較したいときにも使えます。

**試験での見分け方**  
公平性、説明責任、feature attribution、bias という言葉が出たら、Clarify の可能性が高いです。

## 7-2. SageMaker Model Monitor

**役割**  
本番に出したモデルの品質監視です。  
入力分布や予測品質の変化を検知し、ドリフトを見つけます。

**実例**  
去年の顧客属性では高精度だったモデルが、今年の新しい顧客層では精度低下してしまうことがあります。  
そのズレを継続的に見張るのが Model Monitor です。

**試験での見分け方**  
本番ではじめて問題が見つかった、入力データが変わった、性能が徐々に落ちた、といった文脈なら有力候補です。

## 7-3. Model Registry

**役割**  
モデルの版管理、承認状態管理、デプロイ候補の整理を行います。

**実例**  
モデル v1、v2、v3 があり、評価を通過した版だけを本番へ上げたい場面です。  
どのモデルが承認済みで、どの評価結果を持っていたかを整理できます。

## 7-4. Model Cards

**役割**  
モデルの用途、制約、評価結果、リスク、推奨事項などを文書化します。

**実例**  
監査や社内審査で、このモデルは何のために作られ、どのデータで評価され、どんな注意点があるかを説明したいときです。

**試験向けの意味**  
モデルそのものではなく、**説明と記録を残す仕組み** として覚えるとよいです。

## 7-5. Lineage Tracking

**役割**  
このモデルは、どのデータとどの処理から生まれたかを追跡します。

**実例**  
本番障害が起きたときに、どの学習データ、どの処理ジョブ、どのモデルバージョンが使われたかをたどりたい場面です。

## 7-6. SageMaker Pipelines

**役割**  
前処理、学習、評価、登録、配備をワークフローとして自動化します。

**実例**  
新しい学習データが届いたら、前処理して再学習し、評価を通過したら Model Registry に登録し、承認後に本番へ出したい。  
こういう一連の流れに向いています。

## 8. セキュリティと本番運用

### VPC 配置と network isolation

SageMaker AI では、学習ジョブや関連処理を VPC 内のサブネットで動かし、必要に応じて外部との通信を絞れます。  
さらに、network isolation によって外向き通信を制御したいケースもあります。

### 実例

医療データや金融データのように、外部へ出してはいけない情報を学習や評価に使うケースです。  
このときは、単に IAM を厳しくするだけでなく、ネットワーク面の分離も大切になります。

### 試験での見分け方

- 機密データ
- プライベートネットワーク
- インターネット遮断
- 厳格なアクセス制御

このあたりが見えたら、VPC、IAM、暗号化、分離設計を意識します。

---

## 9. 試験での判断フロー

問題を見たら、次の順で整理すると落ち着きやすいです。

### Step 1. Bedrock と SageMaker AI のどちらか

- 既存 FM を素早く使う → Bedrock 寄り
- 学習や微調整、推論基盤、監視、承認まで広く扱う → SageMaker AI 寄り

### Step 2. 何のフェーズか

- データ準備か
- ラベリングか
- モデル選定・微調整か
- 本番推論か
- 監視か
- ガバナンスか

### Step 3. 推論方式を選ぶ

- 即時応答か
- たまにしか使わないか
- 重くて待てるか
- 一括処理か

### Step 4. 本番運用要件を見る

- ドリフト監視が必要か
- 説明責任が必要か
- 承認フローが必要か
- 再現性が必要か

## 10. よくある引っかけ整理

### 引っかけ 1

**既存モデルを使うだけなのに、SageMaker を選びたくなる**

生成 AI 試験では、まず Bedrock が自然なケースがかなりあります。  
わざわざ学習や配備基盤を組まなくてもよいなら、Bedrock の方が素直です。

### 引っかけ 2

**Serverless を万能だと思ってしまう**

断続利用には便利ですが、常に低遅延が必要な本番 UI には向かないことがあります。  
アクセス頻度やレイテンシー要件をよく見ます。

### 引っかけ 3

**Batch Transform と Async を混同する**

- 1件ずつ重い処理を非同期で受ける → Async
- まとまったデータをオフラインで一括処理 → Batch Transform

### 引っかけ 4

**Model Monitor と Clarify を混同する**

- 本番での品質監視、ドリフト検知 → Model Monitor
- バイアス、説明可能性、公平性 → Clarify

### 引っかけ 5

**Model Registry と Model Cards を混同する**

- 版管理、承認状態、モデル群の整理 → Model Registry
- モデルの説明書、監査向け記録 → Model Cards

## 11. 試験直前の超圧縮メモ

- **Bedrock** は既存 FM を素早く安全に使う
- **SageMaker AI** は学習・微調整・配備・監視まで深く制御する
- **Data Wrangler** は表形式データ整備の入口
- **Processing** は前処理や後処理をジョブで回す
- **Ground Truth** は人手ラベリング
- **JumpStart** はモデル選定と微調整の入口
- **Real-time** は即時応答
- **Serverless** は断続利用
- **Async** は重くて待てる処理
- **Batch Transform** は一括オフライン処理
- **Clarify** は公平性・説明可能性
- **Model Monitor** は本番ドリフト監視
- **Model Registry** は版管理と承認
- **Model Cards** は監査向けの説明書
- **Pipelines** は MLOps 自動化
- **Lineage** はどのデータ・処理からできたかの追跡

## 12. 最後に: 覚え方の芯

この領域は、サービス名を丸暗記すると混乱しやすいです。  
次の問いを自分に投げるだけで、だいぶ整理できます。

- 今やりたいのは、使うことか、育てることか
- これはデータの話か、モデルの話か、本番運用の話か
- 人が待つのか、待たないのか
- 監視したいのか、説明したいのか、承認したいのか

この4つで切ると、選択肢の景色がかなりすっきりします。
